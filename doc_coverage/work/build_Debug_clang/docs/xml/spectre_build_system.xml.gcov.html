      35             : <sect2 id="spectre_build_system_1adding_unit_tests">
      36             : <title>Adding Unit Tests</title>
      37             : <para>We use the <ulink url="https://github.com/philsquared/Catch">Catch</ulink> testing framework for unit tests. All unit tests are housed in <computeroutput>tests/Unit</computeroutput> with subdirectories for each subdirectory of <computeroutput>src</computeroutput>. Add the <computeroutput>cpp</computeroutput> file to the appropriate subdirectory and also to the <computeroutput>CMakeLists.txt</computeroutput> in that subdirectory. Inside the source file you can create a new test by adding a <computeroutput>SPECTRE_TEST_CASE(&quot;Unit.Dir.Component&quot;, &quot;[Unit][Dir][Tag]&quot;)</computeroutput>. The <computeroutput>[Tag]</computeroutput> is optional and you can have more than one, but the tags should be used quite sparingly. The purpose of the tags is to be able to run all unit tests or all tests of a particular set of components, e.g. <computeroutput>ctest -L Data</computeroutput> to run all tests inside the <computeroutput>Data</computeroutput> directory. Please see <ref refid="writing_unit_tests" kindref="compound">writing unit tests</ref>, other unit tests and the <ulink url="https://github.com/philsquared/Catch">Catch documentation</ulink> for more help on writing tests. Unit tests should take as short a time as possible, with a goal of less than two seconds. Please also limit the number of distinct cases (by using <computeroutput>SECTION</computeroutput>s).</para><para>You can check the unit test coverage of your code by installing all the optional components and then running <computeroutput>make unit-test-coverage</computeroutput> (after re-running CMake). This will create the directory <computeroutput>BUILD_DIR/docs/html/unit-test-coverage/</computeroutput> which is where the coverage information is located. Open the <computeroutput>index.html</computeroutput> file in your browser and make sure that your tests are indeed checking all lines of your code. Your pull requests might not be merged until your line coverage is over 90% (we are aiming for 100% line coverage wherever possible). Unreachable lines of code can be excluded from coverage analysis by adding the inline comment <computeroutput>LCOV_EXCL_LINE</computeroutput> or a block can be excluded using <computeroutput>LCOV_EXCL_START</computeroutput> and <computeroutput>LCOV_EXCL_STOP</computeroutput>. However, this should be used extremely sparingly since unreachable code paths should be removed from the code base altogether.</para></sect2>

      43             : <para>To add an external dependency, first add a <computeroutput>SetupDEPENDENCY.cmake</computeroutput> file to the <computeroutput>cmake</computeroutput> directory. You should model this after one of the existing ones for <computeroutput>Catch</computeroutput> or <computeroutput>Brigand</computeroutput> if you&apos;re adding a header-only library and <computeroutput>yaml-cpp</computeroutput> if the library is not header-only. If CMake does not already support <computeroutput>find_package</computeroutput> for the library you&apos;re adding you can write your own. These should be modeled after <computeroutput>FindBrigand</computeroutput> or <computeroutput>FindCatch</computeroutput> for header-only libraries, and <computeroutput>FindYAMLCPP</computeroutput> for compiled libraries. Be sure to test both that setting <computeroutput>LIBRARY_ROOT</computeroutput> works correctly for your library, and also that if the library is required that CMake fails gracefully if the library is not found.</para></sect2>

      81             : </para><para><heading level="2">Formaline</heading>
      82             : </para><para>SpECTRE&apos;s implementation of Formaline is based on, but distinct in implementation from, the original design by <ulink url="https://github.com/hypercott/formaline">Erik Schnetter and Christian Ott</ulink>, which embeds an archive of the source tree into the executable. The original design creates a C/C++ file with a function that returns an array/vector of <computeroutput>char</computeroutput>s (a byte stream). However, this results in a very large source file (50MB or more), which is very slow to compile and ends up more than doubling the link time. Instead, SpECTRE&apos;s Formaline implementation uses the linker <computeroutput>ld</computeroutput> to encode a file into an object, which means rather than creating a large source file, we can directly encode the source tree archive into the binary at the linking stage.</para><para>Most of SpECTRE&apos;s Formaline is implemented inside the <computeroutput>tools/WrapLinker.sh</computeroutput> script. Function declarations are provided in <computeroutput><ref refid="Formaline_8hpp_source" kindref="compound">Utilities/Formaline.hpp</ref></computeroutput> and a small function that writes the source file to disk is defined in <computeroutput>Utilities/Formaline.cpp</computeroutput>. The first Formaline-related thing done in <computeroutput>WrapLinker.sh</computeroutput> is to archive everything in the source directory tracked by git. Once the archive is created we run <computeroutput>ld -r -b binary -o object.o src.tar.gz</computeroutput> (with unique names for <computeroutput>object.o</computeroutput> and <computeroutput>src.tar.gz</computeroutput> for each executable that is built to avoid name collisions) to generate an object file with the source file encoded from <computeroutput>_binary_src_tar_gz_start</computeroutput> to <computeroutput>_binary_src_tar_gz_end</computeroutput>. Next we write a C++ source file that defines a function <computeroutput>get_archive</computeroutput> to convert the byte stream into a <computeroutput><ref refid="cpp/container/vector" kindref="compound" external="/work/spectre/docs/config/cppreference-doxygen-web.tag.xml">std::vector</ref>&lt;char&gt;</computeroutput>. We also encode the output of <computeroutput>printenv</computeroutput>, the various <computeroutput>PATH</computeroutput> environment variables, and the CMake generated <computeroutput>LibraryVersions.txt</computeroutput> file into the source file. Finally, the generated source file is built during the linking phase and the object file containing the source archive is linked into the executable.</para><para>To further aid in reproducibility, the <computeroutput>printenv</computeroutput> output and <computeroutput>LibraryVersions.txt</computeroutput> contents are written to HDF5 files as part of the <computeroutput><ref refid="classh5_1_1Header" kindref="compound">h5::Header</ref></computeroutput> object.

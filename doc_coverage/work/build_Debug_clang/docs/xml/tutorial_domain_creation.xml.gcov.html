       7             : <para><heading level="3">Domain Decomposition</heading>
       8             : </para><para>The spatial domain on which the solution is computed is divided into non-overlapping cells. For 2D and 3D domains, we use curvilinear quadrilateral and curvilinear hexahedral elements, respectively. The specification of a particular spatial domain is given by the <ref refid="classDomainCreator" kindref="compound">DomainCreator</ref>. DomainCreators specify the properties of the initial elements (before AMR takes place), which are then used by <ref refid="classDomain" kindref="compound">Domain</ref> to create Blocks. Each <ref refid="classBlock" kindref="compound">Block</ref> holds a CoordinateMap (described below) as well as information about its neighbors. For geometrical domains, (e.g. rectilinear, spherical) there are a few shortcuts that can be used such that a user does not need to specify by hand the map and neighbor information for each Block; these methods are explained below.</para><para><heading level="2">CoordinateMaps</heading>
       9             : </para><para>Each <ref refid="classBlock" kindref="compound">Block</ref> in the <ref refid="classDomain" kindref="compound">Domain</ref> must hold a CoordinateMap which describes how to map the logical cube (a domain that goes [-1. 1] in each dimension) to the curvilinear hexahedral element the Block describes. The CoordinateMap also provides the jacobian of the mapping.</para><para><heading level="3">Shortcuts for CoordinateMaps</heading>
      10             : </para><para>For spherical domains, Wedge3D implements the cubed-sphere map, and there exists the method <computeroutput>wedge_coordinate_maps</computeroutput> in <ref refid="DomainHelpers_8hpp" kindref="compound">Domain/DomainHelpers.hpp</ref> to quickly construct six of these maps at once. For rectilinear multicube domains, Affine and Equiangular map the logical cubes to cartesian cubes. The method <computeroutput>maps_for_rectilinear_domains</computeroutput> in DomainHelpers allows the user to obtain the CoordinateMaps for all of the Blocks in such a domain at once. These can both be used to provide the map arguments to the <ref refid="classDomain" kindref="compound">Domain</ref> constructor.</para><para><heading level="2">Boundary Information</heading>
      11             : </para><para>Each <ref refid="classBlock" kindref="compound">Block</ref> must know which of the other Blocks in the <ref refid="classDomain" kindref="compound">Domain</ref> are its neighbors and which of its logical directions points to an external boundary.</para><para><heading level="3">Shortcuts for Boundary Information</heading>
      12             : </para><para>A quick way to encode the neighbor and boundary information for the blocks in a domain is through our convention of numbering and ordering the corners of the blocks in a well-defined way. For spherical domains, DomainHelpers has the methods <computeroutput>corners_for_radially_layered_domains</computeroutput> and <computeroutput>corners_for_biradially_layered_domains</computeroutput>, which provides the proper corner numberings for the maps obtained from the method <computeroutput>wedge_coordinate_maps</computeroutput>.

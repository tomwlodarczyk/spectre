      11             : <title>What should I know before I get started?</title>
      12             : <para>SpECTRE is being developed in support of our collaborative Simulating eXtreme Spacetimes (SXS) research program into the multi-messenger astrophysics of neutron star mergers, core-collapse supernovae, and gamma-ray bursts. As such, almost all of the current contributors to SpECTRE are members of SXS institutions, and a large amount of discussion about SpECTRE is done in internal SXS meetings. If you are a member of SXS and wish to get involved, please contact one of the project leaders.</para><para>In the future, we hope that SpECTRE can be applied to problems across discipline boundaries, and that it can be a true community code. At the present time, however, SpECTRE cannot yet solve realistic problems, and broad overview documentation is in an early, incomplete stage. Therefore, if you are not a member of SXS, but are interested in contributing to SpECTRE, we strongly encourage you to contact us at <ulink url="mailto:questions@spectre-code.org">questions@spectre-code.org</ulink> to discuss possible contributions.</para></sect2>

      19             : <para>Bugs are tracked as <ulink url="https://guides.github.com/features/issues/">GitHub issues</ulink>.       20             : <listitem><para><bold>Use a clear and descriptive title</bold> for the issue to identify the problem.</para></listitem><listitem><para><bold>Describe the exact steps which reproduce the problem</bold> in as much detail as possible.       21             : </para><para>Provide more context by answering these questions:</para><para><itemizedlist>

      23             : </para><para>Include details about your configuration and environment:</para><para><itemizedlist>
      24             : <listitem><para><bold>Add as an attachment</bold> (or add the contents of) the following:<itemizedlist>
      25             : <listitem><para>The text output by SpECTRE (including any stack trace)</para></listitem><listitem><para>The input file(s)</para></listitem><listitem><para>$SPECTRE_BUILD_DIR/LibraryVersions.txt</para></listitem></itemizedlist>
      26             : </para></listitem><listitem><para><bold>What is the name and version of the OS you&apos;re using</bold>?</para></listitem><listitem><para>If possible (for SXS computers or HPC systems), a <bold>path to a run directory</bold> that is accessible by SpECTRE core developers.</para></listitem></itemizedlist>

      49             : <title>Pull Requests</title>
      50             : <para>Code contributions to SpECTRE follow a <ulink url="https://help.github.com/articles/about-pull-requests/">pull request model</ulink></para><para>The process described here has several goals:</para><para><itemizedlist>
      51             : <listitem><para>Maintain SpECTRE&apos;s code and documentation quality</para></listitem><listitem><para>Reach science goals in a timely manner</para></listitem><listitem><para>Fix problems that are important to users</para></listitem><listitem><para>Engage the community in working toward the best possible code</para></listitem><listitem><para>Enable a sustainable system for SpECTRE&apos;s maintainers to review contributions</para></listitem></itemizedlist>
      52             : </para><para>Please follow these steps to have your contribution considered by the maintainers:</para><para><orderedlist>
      53             : <listitem><para>Follow the <ref refid="code_review_guide" kindref="compound">code review guidelines</ref>, the <ref refid="writing_unit_tests" kindref="compound">guide to writing unit tests</ref>, and the <ref refid="writing_good_dox" kindref="compound">guide to writing documentation</ref></para></listitem><listitem><para>Follow all instructions in the pull request template. Reference related issues and pull requests.</para></listitem><listitem><para>After you submit your pull request, verify that all <ulink url="https://help.github.com/articles/about-status-checks/">status checks</ulink> are passing</para></listitem></orderedlist>
      54             : </para><para><blockquote><para>If a status check is failing, and you believe that the failure is unrelated to your change, please leave a comment on the pull request explaining why you believe the failure is unrelated. If we conclude that the failure was a false positive, then we will open an issue to track that problem with our status check suite. </para></blockquote></para><para>While the prerequisites above must be satisfied prior to having your pull request reviewed, the reviewers may ask you to complete additional design work, tests, or other changes before your pull request can be ultimately accepted.</para></sect2>
      55             : <sect2 id="contributing_to_spectre_1pull-request-reviews">
      56             : <title>How SpECTRE pull request reviews are conducted</title>
      57             : <para><blockquote><para>Note that these are guidelines and not rigid rules. </para></blockquote></para><para>Please make your pull requests as small as reasonably possible, as smaller pull requests are easier to review. In general, longer pull requests take longer to review, with the time scaling exponentially with the number of lines changed. Therefore if your pull request is too large, we may ask you to break it up into several smaller pull requests.</para><para><blockquote><para>Below, days mean business days, so if the time period includes the weekend, add two days, and for major holidays add a day. Furthermore, most of us are academics, and we occasionally go to conferences which may lead to delays in the review process. Also do not expect much to happen between December 20th and January 3rd. </para></blockquote></para><para>Most pull requests submitted to SpECTRE will be reviewed in the following manner:<itemizedlist>
      58             : <listitem><para>When creating the pull request, the author should add the appropriate labels. If the pull request is not a <computeroutput>new design</computeroutput> or <computeroutput>in progress</computeroutput> (discussed below), the author may either assign two reviewers (if GitHub suggested any) or add the <computeroutput>reviewers wanted</computeroutput> label.</para></listitem><listitem><para>Within two days, assigned reviewers should either confirm that they are able to review, or remove themselves while adding the <computeroutput>reviewers wanted</computeroutput> label.</para></listitem><listitem><para>Anyone is welcome to self-assign themselves as a reviewer; when there are at least two reviewers, the <computeroutput>reviewers wanted</computeroutput> label should be removed.</para></listitem><listitem><para>If there are not two reviewers after two days, a <ulink url="#core-developers">SpECTRE core developer</ulink> will assign reviewers, removing the <computeroutput>reviewers wanted</computeroutput> label. If for some reason, no reviewers have been assigned after three days, the author of the pull request should feel free to <ulink url="https://github.blog/2011-03-23-mention-somebody-they-re-notified/">ping the core developers</ulink> (e.g <computeroutput>@sxs-collaboration/spectre-core-devs please assign reviewers</computeroutput>)</para></listitem><listitem><para>Assigned reviewers should submit their review in as timely a manner as possible.</para></listitem><listitem><para>Anyone can request changes within either the first two days of the pull request, or within a day after the initial reviews of the assigned reviewers. After this period, only the assigned reviewers can request changes, unless someone believes the code is wrong. Non-reviewers are allowed to make comments, which the pull request author is encouraged to address. Alternatively the author can create an issue with the suggested changes, assigned to themselves, which would then be addressed in a subsequent pull request by the author.</para></listitem><listitem><para>If any requested change is unclear to the author, they should ping the reviewer and ask for clarification. Authors and reviewers are encouraged to talk to one another (in person, via Google hangout, or some other verbal method if possible) to resolve any issues.</para></listitem><listitem><para>Reviewers are encouraged to <ulink url="https://github.blog/2011-03-23-mention-somebody-they-re-notified/">ping others @GITHUB_USERNAME</ulink> to get opinions on code they are unsure about.</para></listitem><listitem><para>It is permissible to have a group code review led by one of the reviewers. The reviewer should comment on who was present at the group review.</para></listitem><listitem><para>If necessary, pull requests can also be discussed at one of the weekly SpECTRE meetings.</para></listitem><listitem><para>If changes are requested, the author should fix all of them in one or more fixup commits (where the first line of the commit message should begin with <computeroutput>fixup</computeroutput>), and then ping the reviewers that the pull request is updated. In addition the <computeroutput>updated</computeroutput> label can be added.</para></listitem><listitem><para>Once a pull request is updated, the reviewers should either request further changes (removing the <computeroutput>updated</computeroutput> label) or tell the pull request author to squash their commits.</para></listitem><listitem><para>Once all reviewers give the okay to squash, the author should <ulink url="https://git-scm.com/book/en/v2/Git-Tools-Rewriting-History">rebase on develop and then squash their commits</ulink> into one or more self-contained commits (such that the code will compile and pass all tests after each commit). The squashed commits will need to be force pushed.</para></listitem><listitem><para>Once the commits are squashed, the author should ping the reviewers.</para></listitem><listitem><para>Once all reviewers have approved the pull request, someone should ping the <computeroutput>@sxs-collaboration/spectre-core-devs</computeroutput>. If one of the original reviewers is a core developer, this is not necessary and the core developer can merge the pull request.</para></listitem><listitem><para>One of the core developers will perform a final cursory review, requesting changes only for major problems, and commenting on other possible changes.</para></listitem><listitem><para>The pull request author should address all final requested changes and may either also fix final suggested changes, or create an issue with the suggested changes, which will be addressed in a subsequent pull request by the author.</para></listitem><listitem><para>The SpECTRE core developer will merge the pull request once all comments have been addressed, the code passes CI, and all pull requests the pull request depends on have been merged.</para></listitem></itemizedlist>
      59             : </para><para>Critical bug fixes (i.e. the code is broken) can be merged after two expedited reviews by SpECTRE core developers. If necessary, an issue can be created if further changes are desired.</para><para>Pull requests that are designated <computeroutput>new design</computeroutput> are expected to have a longer review period, including discussions during weekly SpECTRE meetings. SpECTRE core developers will provide reasonable review deadlines once the new design is finalized.</para><para>If you would like feedback on a pull request prior to it being ready for formal review, please label it with <computeroutput>in progress</computeroutput> and ping whomever you wish to get feedback from. As long as the <computeroutput>in progress</computeroutput> label remains, no one should review the pull request.</para><sect3 id="contributing_to_spectre_1git-commit-messages">

       9             : # Introduction {#dev_guide_parallelization_introduction}
      10             : 
      11             : SpECTRE builds a layer on top of Charm++ that performs various safety checks and
      12             : initialization for the user that can otherwise lead to difficult-to-debug
      13             : undefined behavior. The central concept is what is called a %Parallel
      14             : Component. A %Parallel Component is a struct with several type aliases that
      15             : is used by SpECTRE to set up the Charm++ chares and allowed communication
      16             : patterns. %Parallel Components are input arguments to the compiler, which then
      17             : writes the parallelization infrastructure that you requested for the executable.
      18             : There is no restriction on the number of %Parallel Components, though
      19             : practically it is best to have around 10 at most.
      20             : 
      21             : Here is an overview of what is described in detail in the sections below:
      22             : 
      23             : - \ref dev_guide_parallelization_metavariables_class "Metavariables": Provides
      24             :   high-level configuration to the compiler, e.g. the physical system to be
      25             :   simulated.
      26             : - \ref dev_guide_parallelization_phases_of_execution "Phases": Defines distinct
      27             :   simulation phases separated by a global synchronization point,
      28             :   e.g. `Initialization`, `Evolve` and `Exit`.
      29             : - \ref dev_guide_parallelization_core_algorithm "Algorithm": In each phase,
      30             :   repeatedly iterates over a list of actions until the current phase ends.
      31             : - \ref dev_guide_parallelization_parallel_components "Parallel component":
      32             :   Maintains and executes its algorithm.
      33             : - \ref dev_guide_parallelization_actions "Action": Performs a computational
      34             :   task, e.g. evaluating the right hand side of the time evolution equations. May
      35             :   require data to be received from another action potentially being executed on
      36             :   a different core or node.
      37             : 
      38             : # The Metavariables Class {#dev_guide_parallelization_metavariables_class}
      39             : 
      40             : SpECTRE takes a different approach to input options passed to an executable than
      41             : is common. SpECTRE not only reads an input file at runtime but also has many
      42             : choices made at compile time. The compile time options are specified by what is
      43             : referred to as the metavariables. What exactly the metavariables struct
      44             : specifies depends on the executable, but all metavariables structs must
      45             : specify the following:
      46             : 
      47             : - `help`: a `static constexpr OptionString` that will be printed as part of the
      48             :   help message. It should describe the executable and basic usage of it, as well
      49             :   as any non-standard options that must be specified in the metavariables and
      50             :   their current values. An example of a help string for one of the testing
      51             :   executables is:
      52             :   \snippet Test_AlgorithmCore.cpp help_string_example
      53             : - `component_list`: a `tmpl::list` of the parallel components (described below)
      54             :   that are to be created. Most evolution executables will have the
      55             :   `DgElementArray` parallel component listed. An example of a `component_list`
      56             :   for one of the test executables is:
      57             :   \snippet Test_AlgorithmCore.cpp component_list_example
      58             : - `using const_global_cache_tags`: a `tmpl::list` of tags that are
      59             :   used to place items in the ConstGlobalCache.  The alias may be
      60             :   omitted if the list is empty.
      61             : - `Phase`: an `enum class` that must contain at least `Initialization` and
      62             :   `Exit`. Phases are described in the next section.
      63             : - `determine_next_phase`: a static function with the signature
      64             :   \code
      65             :     static Phase determine_next_phase(
      66             :       const Phase& current_phase,
      67             :       const Parallel::CProxy_ConstGlobalCache<EvolutionMetavars>& cache_proxy)
      68             :       noexcept;
      69             :   \endcode
      70             :   What this function does is described below in the discussion of
      71             :   \ref dev_guide_parallelization_phases_of_execution "phases".
      72             : 
      73             : There are also several optional members:
      74             : 
      75             : - `input_file`: a `static constexpr OptionString` that is the default name of
      76             :   the input file that is to be read. This can be overridden at runtime by
      77             :   passing the `--input-file` argument to the executable.
      78             : - `ignore_unrecognized_command_line_options`: a `static constexpr bool` that
      79             :   defaults to `false`. If set to `true` then unrecognized command line options
      80             :   are ignored. Ignoring unrecognized options is generally only necessary for
      81             :   tests where arguments for the testing framework,
      82             :   [Catch](https://github.com/catchorg/Catch2/), are passed to the executable.
      83             : 
      84             : # Phases of an Execution {#dev_guide_parallelization_phases_of_execution}
      85             : 
      86             : Global synchronization points, where all cores wait for each other, are
      87             : undesirable for scalability reasons. However, they are sometimes inevitable for
      88             : algorithmic reasons. That is, in order to actually get a correct solution you
      89             : need to have a global synchronization. SpECTRE executables can have multiple
      90             : phases, where after each phase a global synchronization occurs. By global
      91             : synchronization we mean that no parallel components are executing or have more
      92             : tasks to execute: everything is waiting to be told what tasks to perform next.
      93             : 
      94             : Every executable must have at least two phases, `Initialization` and
      95             : `Exit`. The next phase is decided by the static member function
      96             : `determine_next_phase` in the metavariables. Currently this function has access
      97             : to the phase that is
      98             : ending, and also the global cache. In the future we will add support for
      99             : receiving data from various components to allow for more complex decision
     100             : making. Here is an example of a `determine_next_phase` function and the `Phase`
     101             : enum class:
     102             : \snippet Test_AlgorithmCore.cpp determine_next_phase_example
     103             : 
     104             : In contrast, an evolution executable might have phases
     105             : `Initialization`, `SetInitialData`, `Evolve`, and `Exit`, but have a
     106             : similar `switch` or `if-else` logic in the `determine_next_phase`
     107             : function. The first phase that is entered is always
     108             : `Initialization`. During the `Initialization` phase the
     109             : `Parallel::ConstGlobalCache` is created, all non-array components are created,
     110             : and empty array components are created.  Next, the function
     111             : `allocate_array_components_and_execute_initialization_phase` is called
     112             : which allocates the elements of each array component, and then starts
     113             : the `Initialization` phase on all parallel components. Once all
     114             : parallel components' `Initialization` phase is complete, the next
     115             : phase is determined and the `execute_next_phase` function is called on
     116             : all the parallel components.
     117             : 
     118             : At the end of an execution the `Exit` phase has the executable wait to make sure
     119             : no parallel components are performing or need to perform any more tasks, and
     120             : then exits. An example where this approach is important is if we are done
     121             : evolving a system but still need to write data to disk. We do not want to exit
     122             : the simulation until all data has been written to disk, even though we've
     123             : reached the final time of the evolution.
     124             : 
     125             : \warning Currently dead-locks are treated as successful termination. In the
     126             : future checks against deadlocks will be performed before terminating.
     127             : 
     128             : # The Algorithm {#dev_guide_parallelization_core_algorithm}
     129             : 
     130             : Since most numerical algorithms repeat steps until some criterion such as the
     131             : final time or convergence is met, SpECTRE's parallel components are designed to
     132             : do such iterations for the user. An Algorithm executes an ordered list of
     133             : actions until one of the actions cannot be evaluated, typically because it is
     134             : waiting on data from elsewhere. When an algorithm can no longer evaluate actions
     135             : it passively waits by handing control back to Charm++. Once an algorithm
     136             : receives data, typically done by having another parallel component call the
     137             : `receive_data` function, the algorithm will try again to execute the next
     138             : action. If the algorithm is still waiting on more data then the algorithm will
     139             : again return control to Charm++ and passively wait for more data. This is
     140             : repeated until all required data is available. The actions that are iterated
     141             : over by the algorithm are called iterable actions and are described below.
     142             : Since the action list is phase dependent we refer to them generally as
     143             : phase-dependent action lists (PDALs, pronounced "pedals").
     144             : 
     145             : # Parallel Components {#dev_guide_parallelization_parallel_components}
     146             : 
     147             : Each %Parallel Component struct must have the following type aliases:
     148             : 1. `using chare_type` is set to one of:
     149             :    1. `Parallel::Algorithms::Singleton`s have one object in the entire execution
     150             :       of the program.
     151             :    2. `Parallel::Algorithms::Array`s hold zero or more elements, each of which
     152             :       is an object distributed to some core. An array can grow and shrink in
     153             :       size dynamically if need be and can also be bound to another array. A
     154             :       bound array has the same number of elements as the array it is bound to,
     155             :       and elements with the same ID are on the same core. See Charm++'s chare
     156             :       arrays for details.
     157             :    3. `Parallel::Algorithms::Group`s are arrays with
     158             :       one element per core which are not able to be moved around between
     159             :       cores. These are typically useful for gathering data from array elements
     160             :       on their core, and then processing or reducing the data further. See
     161             :       [Charm++'s](http://charm.cs.illinois.edu/help) group chares for details.
     162             :    4. `Parallel::Algorithms::Nodegroup`s are similar to
     163             :       groups except that there is one element per node. For Charm++ SMP (shared
     164             :       memory parallelism) builds, a node corresponds to the usual definition of
     165             :       a node on a supercomputer. However, for non-SMP builds nodes and cores are
     166             :       equivalent. We ensure that all entry method calls done through the
     167             :       Algorithm's `simple_action` and `receive_data` functions are
     168             :       threadsafe. User controlled threading is possible by calling the non-entry
     169             :       method member function `threaded_action`.
     170             : 2. `using metavariables` is set to the Metavariables struct that stores the
     171             :    global metavariables. It is often easiest to have the %Parallel
     172             :    Component struct have a template parameter `Metavariables` that is the
     173             :    global metavariables struct. Examples of this technique are given below.
     174             : 3. `using phase_dependent_action_list` is set to a `tmpl::list` of
     175             :    `Parallel::PhaseActions<PhaseType, Phase, tmpl::list<Actions...>>`
     176             :    where each `PhaseAction` represents a PDAL that will be executed on
     177             :    the parallel component during the specified phase. The %Actions are
     178             :    executed in the order that they are given in the `tmpl::list`s of
     179             :    the PDALs, but the phases need not be run in linear order. However,
     180             :    `db::DataBox` types are constructed assuming the phases are
     181             :    performed from first in the `phase_dependent_action_list` to the
     182             :    last. Simple actions (described below) can be executed in any
     183             :    phase. If there are no iterable actions in a phase then a
     184             :    `PhaseAction` need not be specified for that phase. However, at
     185             :    least one `PhaseAction`, even if it is empty, must be specified.
     186             : 4. `using initialization_tags` which is a `tmpl::list` of all the tags
     187             :    that will be inserted into the initial `db::DataBox` of each component.
     188             :    These tags are db::SimpleTag%s that have have a `using option_tags`
     189             :    type alias and a static function `create_from_options` (see the
     190             :    example below).  This list can usually be constructed from the
     191             :    initialization actions of the component (i.e. the list of actions
     192             :    in the `PhaseAction` list for the `Initialization` phase) using the
     193             :    helper function `Parallel::get_initialization_tags` (see the
     194             :    examples of components below).  Each initialization action may
     195             :    specify a type alias `using initialization_tags` which are a
     196             :    `tmpl::list` of tags that will be fetched from the db::DataBox by the
     197             :    action.  All `initialization_tags` are removed from the db::DataBox of
     198             :    the component at the end of the `Initialization` phase, except for
     199             :    tags listed in a type alias `using initialization_tags_to_keep` that
     200             :    may appear in each initialization action.
     201             : 5. `using const_global_cache_tags` is set to a `tmpl::list` of tags
     202             :    that are required by the `allocate_array` function of an array
     203             :    component, or simple actions called on the parallel component.
     204             :    These tags correspond to items that are stored in the
     205             :    Parallel::ConstGlobalCache (of which there is one copy per Charm++
     206             :    node).  The alias can be omitted if the list is empty.  (See
     207             :    `array_allocation_tags` below for specifying tags needed for the
     208             :    `allocate_array` function, but will not be added to the
     209             :    Parallel::ConstGlobalCache.)
     210             : 
     211             : \note Array parallel components must also specify the type alias `using
     212             : array_index`, which is set to the type that indexes the %Parallel Component
     213             : Array. Charm++ allows arrays to be 1 through 6 dimensional or be indexed by a
     214             : custom type. The Charm++ provided indexes are wrapped as
     215             : `Parallel::ArrayIndex1D` through `Parallel::ArrayIndex6D`. When writing custom
     216             : array indices, the [Charm++ manual](http://charm.cs.illinois.edu/help) tells you
     217             : to write your own `CkArrayIndex`, but we have written a general implementation
     218             : that provides this functionality (see `Parallel::ArrayIndex`); all that you need
     219             : to provide is a plain-old-data
     220             : ([POD](http://en.cppreference.com/w/cpp/concept/PODType)) struct of the size of
     221             : at most 3 integers.
     222             : 
     223             : %Parallel array components have a static `allocate_array` function
     224             : that is used to construct the elements of the array. The
     225             : signature of the `allocate_array` functions must be:
     226             : \code
     227             : static void allocate_array(
     228             :     Parallel::CProxy_ConstGlobalCache<metavariables>& global_cache,
     229             :     const tuples::tagged_tuple_from_typelist<initialization_tags>&
     230             :     initialization_items) noexcept;
     231             : \endcode
     232             : The `allocate_array` function is called by the Main parallel component
     233             : when the execution starts and will typically insert elements into
     234             : array parallel components. If the `allocate_array` function depends
     235             : upon input options, the array component must specify a `using
     236             : array_allocation_tags` type alias that is a `tmpl::list` of tags which
     237             : are db::SimpleTag%s that have have a `using option_tags` type alias
     238             : and a static function `create_from_options`. An example is:
     239             : \snippet DistributedLinearSolverAlgorithmTestHelpers.hpp array_allocation_tag
     240             : 
     241             : The `allocate_array` functions of different
     242             : array components are called in random order and so it is not safe to
     243             : have them depend on each other.
     244             : 
     245             : Each parallel component must also decide what to do in the different phases of
     246             : the execution. This is controlled by an `execute_next_phase` function with
     247             : signature:
     248             : \code
     249             : static void execute_next_phase(
     250             :     const typename metavariables::Phase next_phase,
     251             :     const Parallel::CProxy_ConstGlobalCache<metavariables>& global_cache);
     252             : \endcode
     253             : The `determine_next_phase` function in the Metavariables determines the next
     254             : phase, after which the `execute_next_phase` function gets called. The
     255             : `execute_next_phase` function determines what the parallel component should do
     256             : during the next phase. Typically the `execute_next_phase` function should just
     257             : call `start_phase(phase)` on the parallel component. In the future
     258             : `execute_next_phase` may be removed.
     259             : 
     260             : An example of a singleton parallel component is:
     261             : \snippet Test_AlgorithmParallel.cpp singleton_parallel_component
     262             : 
     263             : An example of an array parallel component is:
     264             : \snippet Test_AlgorithmParallel.cpp array_parallel_component
     265             : Elements are inserted into the array by using the Charm++ `insert` member
     266             : function of the CProxy for the array. The `insert` function is documented in
     267             : the Charm++ manual. In the above Array example `array_proxy` is a `CProxy` and
     268             : so all the documentation for Charm++ array proxies applies. SpECTRE always
     269             : creates empty arrays with the constructor and requires users to insert however
     270             : many elements they want and on which cores they want them to be placed. Note
     271             : that load balancing calls may result in array elements being moved.
     272             : 
     273             : # Actions {#dev_guide_parallelization_actions}
     274             : 
     275             : For those familiar with Charm++, actions should be thought of as effectively
     276             : being entry methods. They are functions that can be invoked on a remote object
     277             : (chare/parallel component) using a `CProxy` (see the [Charm++
     278             : manual](http://charm.cs.illinois.edu/help)), which is retrieved from the
     279             : Parallel::ConstGlobalCache using the parallel component struct and the
     280             : `Parallel::get_parallel_component()` function. %Actions are structs with a
     281             : static `apply` method and come in three variants: simple actions, iterable
     282             : actions, and reduction actions. One important thing to note
     283             : is that actions cannot return any data to the caller of the remote method.
     284             : Instead, "returning" data must be done via callbacks or a callback-like
     285             : mechanism.
     286             : 
     287             : The simplest signature of an `apply` method is for iterable actions:
     288             : \snippet Test_AlgorithmCore.cpp apply_iterative
     289             : The return type is discussed at the end of each section describing a particular
     290             : type of action. Simple actions can have additional arguments, do not receive
     291             : the `inboxes` or `ActionList`, and take the `ParallelComponent` as an
     292             : explicit first template parameter. Reduction actions have the same signature as
     293             : simple actions except that the additional arguments must be of the types reduced
     294             : over.
     295             : The `db::DataBox` should be thought of as the member data of the parallel
     296             : component while the actions are the member functions. The combination of a
     297             : `db::DataBox` and actions allows building up classes with arbitrary member data
     298             : and methods using template parameters and invocation of actions. This approach
     299             : allows us to eliminate the need for users to work with Charm++'s interface
     300             : files, which can be error prone and difficult to use.
     301             : 
     302             : The Parallel::ConstGlobalCache is passed to each action so that the
     303             : action has access to global data and is able to invoke actions on
     304             : other parallel components. The `ParallelComponent` template parameter
     305             : is the tag of the parallel component that invoked the action. A proxy
     306             : to the calling parallel component can then be retrieved from the
     307             : Parallel::ConstGlobalCache. The remote entry method invocations are
     308             : slightly different for different types of actions, so they will be
     309             : discussed below. However, one thing that is disallowed for all actions
     310             : is calling an action locally from within an action on the same
     311             : parallel component.  Specifically,
     312             : 
     313             : \snippet Test_AlgorithmNestedApply1.cpp bad_recursive_call
     314             : 
     315             : Here `ckLocal()` is a Charm++ provided method that returns a pointer
     316             : to the local (currently executing) parallel component. See the
     317             : [Charm++ manual](http://charm.cs.illinois.edu/help) for more
     318             : information.  However, you are able to queue a new action to be
     319             : executed later on the same parallel component by getting your own
     320             : parallel component from the Parallel::ConstGlobalCache
     321             : (`Parallel::get_parallel_component<ParallelComponent>(cache)`).  The
     322             : difference between the two calls is that by calling an action through
     323             : the parallel component you will first finish the series of actions you
     324             : are in, then when they are complete Charm++ will call the next queued
     325             : action.
     326             : 
     327             : Array, group, and nodegroup parallel components can have actions invoked in two
     328             : ways. First is a broadcast where the action is called on all elements of the
     329             : array:
     330             : 
     331             : \snippet Test_AlgorithmParallel.cpp broadcast_to_group
     332             : 
     333             : The second case is invoking an action on a specific array element by using the
     334             : array element's index. The below example shows how a broadcast would be done
     335             : manually by looping over all elements in the array:
     336             : 
     337             : \snippet Test_AlgorithmParallel.cpp call_on_indexed_array
     338             : 
     339             : Note that in general you will not know what all the elements in the array are
     340             : and so a broadcast is the correct method of sending data to or invoking an
     341             : action on all elements of an array parallel component.
     342             : 
     343             : The `array_index` argument passed to all `apply` methods is the index into the
     344             : parallel component array. If the parallel component is not an array the value
     345             : and type of `array_index` is implementation defined and cannot be relied on. The
     346             : `ActionList` type is the `tmpl::list` of iterable actions in the current phase.
     347             : That is, it is equal to the `action_list` type alias in the current PDAL.
     348             : 
     349             : ## 1. Simple Actions {#dev_guide_parallelization_simple_actions}
     350             : 
     351             : Simple actions are designed to be called in a similar fashion to
     352             : member functions of classes. They are the direct analog of entry
     353             : methods in Charm++ except that the member data is stored in the
     354             : `db::DataBox` that is passed in as the first argument. A simple action
     355             : must return void but can use `db::mutate` to change values of items in
     356             : the `db::DataBox` if the `db::DataBox` is taken as a non-const
     357             : reference. In some cases you will need specific items to be in the
     358             : `db::DataBox` otherwise the action won't compile. To restrict which
     359             : `db::DataBox`es can be passed you should use `Requires` in the
     360             : action's `apply` function template parameter list. For example,
     361             : \snippet Test_AlgorithmCore.cpp requires_action
     362             : where the conditional checks if any element in the parameter pack `DbTags` is
     363             : `CountActionsCalled`.
     364             : 
     365             : A simple action that does not take any arguments can be called using a `CProxy`
     366             : from the Parallel::ConstGlobalCache as follows:
     367             : 
     368             : \snippet Test_AlgorithmCore.cpp simple_action_call
     369             : 
     370             : If the simple action takes arguments then the arguments must be as follows:
     371             : 
     372             : \snippet Test_AlgorithmNodelock.cpp simple_action_with_args
     373             : 
     374             : ## 2. Iterable Actions {#dev_guide_parallelization_iterable_actions}
     375             : 
     376             : %Actions in the algorithm that are part of the current PDAL are executed one
     377             : after the other until one of them cannot be evaluated. Iterable
     378             : actions may have an `is_ready` method that returns `true` or `false` depending
     379             : on whether or not the action is ready to be evaluated. If no `is_ready` method
     380             : is provided then the action is assumed to be ready to be evaluated. The
     381             : `is_ready` method typically checks that required data from other parallel
     382             : components has been received. For example, it may check that all data from
     383             : neighboring elements has arrived to be able to continue integrating in time.
     384             : The signature of an `is_ready` method must be:
     385             : 
     386             : \snippet Test_AlgorithmCore.cpp is_ready_example
     387             : 
     388             : The `inboxes` is a collection of the tags passed to `receive_data` and are
     389             : specified in the iterable actions member type alias `inbox_tags`, which must be
     390             : a `tmpl::list`. The `inbox_tags` must have two member type aliases, a
     391             : `temporal_id` which is used to identify when the data was sent, and a `type`
     392             : which is the type of the data to be stored in the `inboxes`. The types are
     393             : typically a `std::unordered_map<temporal_id, DATA>`. In the discussed scenario
     394             : of waiting for neighboring elements to send their data the `DATA` type would be
     395             : a `std::unordered_map<TheElementIndex, DataSent>`. Having `DATA` be a
     396             : `std::unordered_multiset` is currently also supported. Here is an example of a
     397             : receive tag:
     398             : 
     399             : \snippet Test_AlgorithmParallel.cpp int_receive_tag
     400             : 
     401             : The `inbox_tags` type alias for the action is:
     402             : 
     403             : \snippet Test_AlgorithmParallel.cpp int_receive_tag_list
     404             : 
     405             : and the `is_ready` function is:
     406             : 
     407             : \snippet Test_AlgorithmParallel.cpp int_receive_tag_is_ready
     408             : 
     409             : Once all of the `int`s have been received, the iterable action is executed, not
     410             : before.
     411             : 
     412             : \warning
     413             : It is the responsibility of the iterable action to remove data from the inboxes
     414             : that will no longer be needed. The removal of unneeded data should be done in
     415             : the `apply` function.
     416             : 
     417             : Iterable actions can change the type of the db::DataBox by adding or
     418             : removing elements/tags from the db::DataBox. The only requirement is
     419             : that the last action in each PDAL returns a db::DataBox that is the
     420             : same type for each iteration. Iterable actions can also request that
     421             : the algorithm no longer be executed, and control which action in the
     422             : current PDAL will be executed next. This is all done via the return
     423             : value from the `apply` function.  The `apply` function for iterable
     424             : actions must return a `std::tuple` of one, two, or three elements. The
     425             : first element of the tuple is the new db::DataBox, which can be the
     426             : same as the type passed in or a db::DataBox with different tags.  Most
     427             : iterable actions will simply return:
     428             : 
     429             : \snippet Test_AlgorithmParallel.cpp return_forward_as_tuple
     430             : 
     431             : By returning the db::DataBox as a reference in a `std::tuple` we avoid
     432             : any unnecessary copying of the db::DataBox. The second argument is an
     433             : optional bool, and controls whether or not the algorithm is
     434             : terminated. If the bool is `true` then the algorithm is terminated, by
     435             : default it is `false`. Here is an example of how to return a
     436             : db::DataBox with the same type that is passed in and also terminate
     437             : the algorithm:

     441             : Notice that we again return a reference to the db::DataBox, which is
     442             : done to avoid any copying. After an algorithm has been terminated it
     443             : can be restarted by passing `false` to the `set_terminate` method or
     444             : by calling `receive_data(..., true)`. Since the order in which
     445             : messages are received is undefined in most cases the
     446             : `receive_data(..., true)` call should be used to restart the
     447             : algorithm.

     449             : The third optional element in the returned `std::tuple` is a `size_t` whose
     450             : value corresponds to the index of the action to be called next in the
     451             : PDAL. The metafunction `tmpl::index_of<list, element>` can be used to
     452             : get an `tmpl::integral_constant` with the value of the index of the element
     453             : `element` in the typelist `list`. For example,

     456             :

     458             : termination `bool` and next action `size_t` are returned by value. The
     459             : metafunction call `tmpl::index_of<ActionList,
     460             : iterate_increment_int0>::%value` returns a `size_t` whose value is
     461             : that of the action `iterate_increment_int0` in the PDAL.  The indexing
     462             : of actions in the PDAL starts at `0`.

     477             : was terminated.

     480             : on an entire array or singleton (this does a broadcast), or an on individual
     481             : element of the array. Here is an example of a broadcast call:
     482             :

     489             : The `receive_data` function always takes a `ReceiveTag`, which is set in the
     490             : actions `inbox_tags` type alias as described above.  The first argument is the
     491             : temporal identifier, and the second is the data to be sent.
     492             : 
     493             : Normally when remote functions are invoked they go through the Charm++ runtime
     494             : system, which adds some overhead. The `receive_data` function tries to elide
     495             : the call to the Charm++ RTS for calls into array components. Charm++ refers to
     496             : these types of remote calls as "inline entry methods". With the Charm++ method
     497             : of eliding the RTS, the code becomes susceptible to stack overflows because
     498             : of infinite recursion. The `receive_data` function is limited to at most 64 RTS
     499             : elided calls, though in practice reaching this limit is rare. When the limit is
     500             : reached the remote method invocation is done through the RTS instead of being
     501             : elided.

     526             : It is often necessary to reduce custom data types, such as `std::vector` or
     527             : `std::unordered_map`. Charm++ supports such custom reductions, and so does our
     528             : layer on top of Charm++.
     529             : Custom reductions require one additional step to calling
     530             : `contribute_to_reduction`, which is writing a reduction function to reduce the
     531             : custom data. We provide a generic type that can be used in custom reductions,
     532             : `Parallel::ReductionData`, which takes a series of `Parallel::ReductionDatum` as
     533             : template parameters and `ReductionDatum::value_type`s as the arguments to the
     534             : constructor. Each `ReductionDatum` takes up to four template parameters (two
     535             : are required). The first is the type of data to reduce, and the second is a
     536             : binary invokable that is called at each step of the reduction to combine two
     537             : messages. The last two template parameters are used after the reduction has
     538             : completed. The third parameter is an n-ary invokable that is called once the
     539             : reduction is complete, whose first argument is the result of the reduction. The
     540             : additional arguments can be any `ReductionDatum::value_type` in the
     541             : `ReductionData` that are before the current one. The fourth template parameter
     542             : of `ReductionDatum` is used to specify which data should be passed. It is a
     543             : `std::index_sequence` indexing into the `ReductionData`.
     544             :

     546             : 
     547             : \snippet Test_AlgorithmReduction.cpp custom_reduction_action
     548             : 
     549             : Note that it takes objects of the types that the reduction was done over as
     550             : additional arguments.

     556             : behavior if the contribute calls are made in different orders on different array

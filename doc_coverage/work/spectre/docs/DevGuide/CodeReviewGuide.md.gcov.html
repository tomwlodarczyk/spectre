       5             : # Code Review Guide {#code_review_guide}
       6             : 
       7             : Code must follow the
       8             : <a href="https://github.com/isocpp/CppCoreGuidelines/blob/master/CppCoreGuidelines.md">C++ Core Guidelines</a>
       9             : and the [Google style guide](https://google.github.io/styleguide/cppguide.html).
      10             : If the Google style guide disagrees with the Core Guidelines, follow the Core
      11             : Guidelines.
      12             : 
      13             : Here we summarize what we view as the more important portions of the guides.
      14             : 
      15             : Stylistic Items:
      16             : 
      17             : * Adhere to [Google style](https://google.github.io/styleguide/cppguide.html).
      18             :   [Can use `clang-format -style=google`]
      19             :   (http://clang.llvm.org/docs/ClangFormat.html).
      20             : * CamelCase: class names template parameters, file names, and directory names.
      21             : * snake_case: function, variable, metafunction and metavariable names.
      22             : * SCREAMING_SNAKE_CASE: macros.
      23             : * Functions or classes only used internally in a library should be in a
      24             :   namespace named `LibraryOrFile_detail`. For example, `databox_detail`,
      25             :   `Tensor_detail` or `ConstantExpressions_detail`.
      26             : * Name unused function parameters `/*parameter_name*/` or `/*meta*/` for TMP
      27             :   cases
      28             : * Type aliases that wrap type traits have a trailing `_t` following the STL
      29             : * Private member variables have a [trailing underscore]
      30             :   (https://google.github.io/styleguide/cppguide.html#Variable_Names).
      31             : * Do not use
      32             :   [Hungarian notation](https://en.wikipedia.org/wiki/Hungarian_notation),
      33             :   e.g. `double* pd_blah` is bad
      34             : * Header order:
      35             :   1. (If a test:) `tests/Unit/TestingFramework.hpp`, followed by one blank line
      36             :   2. (If a cpp file with a corresponding hpp file:) hpp corresponding to cpp,
      37             :      followed by one blank line
      38             :   3. STL and externals (in alphabetical order)
      39             :   4. Blank line
      40             :   5. SpECTRE includes (in alphabetical order)
      41             : * Template definitions in header files are separated from the declaration of
      42             :   the class by the following line, which contains exactly 64 equal signs
      43             : 
      44             : ``` cpp
      45             : // ================================================================
      46             : ```
      47             : 
      48             : * File lists in CMake are alphabetical.
      49             : * No blank lines surrounding Doxygen group comments
      50             :   (<code>// \@{</code> and <code>// \@}</code>).
      51             : * Use the [alternative tokens]
      52             :   (http://en.cppreference.com/w/cpp/language/operator_alternative)
      53             :   `or`, `and`, and `not` instead of `||`, `&&`, and `!`.
      54             : * Use C-style Doxygen comments (`/*! ... */`) when using multi-line math,
      55             :   otherwise C-style and C++ style comments are accepted.
      56             : * When addressing requests on a PR, the commit message must start with
      57             :   `fixup` followed by a descriptive message.
      58             : 
      59             : Code Quality Items:
      60             : 
      61             : * All code passes Clang and CppCheck static analyzers. For help
      62             :   with these tools see \ref static_analysis_tools "here".
      63             : * Almost always `auto`, except with expression templates, i.e. `DataVector`
      64             : * All loops and if statements use braces.
      65             : * Order of declaration in classes is `public` before `private` and member
      66             :   functions before data. See
      67             :   [Google](https://google.github.io/styleguide/cppguide.html#Declaration_Order).
      68             : * Prefer return by value over pass-by-mutable-reference except when mutable
      69             :   reference provides superior performance (in practice if you need a mutable
      70             :   reference use `const gsl::not_null<Type*>` instead of `Type&`). The mutable
      71             :   references must be the first arguments passed to the function.
      72             : * All commits for performance changes provide quantitative evidence and the
      73             :   tests used to obtain said evidence.
      74             : * Never include `<iostream>`, use `Parallel::printf` inside
      75             :   `Parallel/Printf.hpp` instead, which is safe to use in parallel.
      76             : * When using charm++ nodelocks include `<converse.h>` instead of `<lrtslock.h>`.
      77             : * Do not add anything to [the `std` namespace]
      78             :   (http://en.cppreference.com/w/cpp/language/extending_std).
      79             : * Virtual functions are explicitly overridden using the `override` keyword.
      80             : * `pragma once` is to be used for header guards
      81             : * Prefer range-based for loops
      82             : * Use `size_t` for positive integers rather than `int`, specifically when
      83             :   looping over containers. This is in compliance with what the STL uses.
      84             : * Error messages should be helpful. An example of a bad error message is "Size
      85             :   mismatch". Instead this message could read "The number of grid points in the
      86             :   matrix 'F' is not the same as the number of grid points in the determinant.",
      87             :   along with the runtime values of the mentioned quantities if applicable.
      88             : * Mark immutable objects as `const`
      89             : * Make classes serializable by writing a `pup` function
      90             : * If a class stores an object passed into a constructor the object should
      91             :   be taken by-value and `std::move`d into the member variable.
      92             : * Definitions of function and class templates should be in `.cpp` files with
      93             :   explicit instantiations whenever possible. The macro
      94             :   `GENERATE_EXPLICIT_INSTANTIATIONS` is useful for generating many
      95             :   explicit instantiations.
      96             : * Explicit instantiations of functions marked as noexcept should be marked
      97             :   as noexcept as well.
      98             : * Functions that do not throw should be marked `noexcept`. If you're unsure
      99             :   and the function does not use an `OptionContext` or is generating python
     100             :   bindings, mark it `noexcept`.
     101             : * Variable names in macros must avoid name collisions, e.g. inside the
     102             :   `PARSE_ERROR` macro you would write
     103             :   `double variable_name_avoid_name_collisions_PARSE_ERROR = 0.0;`
     104             : * Avoid macros if possible. Prefer `constexpr` functions, constexpr
     105             :   variables, and/or template metaprogramming
     106             : * Explicitly specify `double`s, e.g. `sqrt(2.)` or `sqrt(2.0)` instead of
     107             :   `sqrt(2)`.
     108             : * When the index of a `Tensor` is known at compile time, use
     109             :   `get<a, b>(tensor)` instead of the member `get` function.
     110             : * All necessary header files must be included. In header files, prefer
     111             :   forward declarations if class definitions aren't necessary.
     112             : * Explicitly make numbers floating point, e.g. `2.` or `2.0` over `2`.

       5             : # Writing Python Bindings {#spectre_writing_python_bindings}
       6             : 
       7             : ## CMake and Directory Layout
       8             : 
       9             : To allow users to analyze output from simulations and take advantage of
      10             : SpECTRE's data structures and functions in python, bindings must sometimes be
      11             : written. SpECTRE uses
      12             : [Boost.Python]
      13             : (https://www.boost.org/doc/libs/1_68_0/libs/python/doc/html/index.html)
      14             : to aid with generating the bindings. The C++ code for the bindings should
      15             : generally go in a `Python` subdirectory. For example, the bindings for the
      16             : DataStructures library would go in `src/DataStructures/Python/`. SpECTRE
      17             : provides the `spectre_python_add_module` CMake function to make adding a new
      18             : python module, be it with or without bindings, easy.  The python bindings are
      19             : built only if `-D BUILD_PYTHON_BINDINGS=ON` is passed when invoking cmake.
      20             : 
      21             : The function `spectre_python_add_module` takes as its first argument the module,
      22             : in our case
      23             : `DataStructures`. The bindings library will be prefixed with `Py`, and therefore
      24             : be called `PyDataStructures`. Optionally, a list of `SOURCES` can be passed to
      25             : the CMake function. If the python module will only consist of python files, then
      26             : the `SOURCES` option should not be specified. Python files that should be part
      27             : of the module can be passed with the keyword `PYTHON_FILES`, e.g.
      28             : `PYTHON_FILES Hello.py HelloWorld.py`. Finally, the `MODULE_PATH` named argument
      29             : can be passed with a string that is the path to where the module should be. For
      30             : example, `MODULE_PATH "submodule0/submodule1/"` would mean the module is
      31             : accessed from python using `import spectre.submodule0.submodule1.MODULE_NAME`.
      32             : 
      33             : Here is a complete example of how to call the `spectre_python_add_module`
      34             : function:
      35             : 
      36             : \code
      37             : spectre_python_add_module(
      38             :   ExtraDataStructures
      39             :   MODULE_PATH "DataStructures/"
      40             :   SOURCES Bindings.cpp MyCoolDataStructure.cpp
      41             :   PYTHON_FILES CoolPythonDataStructure.py
      42             :   )
      43             : \endcode
      44             : 
      45             : The library that is added has the name `PyExtraDataStructures` and requires that
      46             : at least the `${SPECTRE_LINK_PYBINDINGS}` be set as the last entry to the
      47             : `target_link_libraries` call. For example,
      48             : 
      49             : \code
      50             : target_link_libraries(
      51             :   PyExtraDataStructures
      52             :   PUBLIC ExtraDataStructures
      53             :   ${SPECTRE_LINK_PYBINDINGS}
      54             :   )
      55             : \endcode
      56             : 
      57             : \warning `${SPECTRE_LINK_PYBINDINGS}` must be the *last* argument to
      58             : `target_link_libraries`.
      59             : 
      60             : ## Writing Bindings
      61             : 
      62             : Once a python module has been added you can write the actual bindings. You
      63             : should structure your bindings directory to reflect the structure of the library
      64             : you're writing bindings for. For example, say we want bindings for `DataVector`
      65             : and `Matrix` then we should have one source file for each class's bindings
      66             : inside `src/DataStructures/Python`. The functions that generate the bindings
      67             : should be in the `py_bindings` namespace and have a reasonable name such as
      68             : `bind_datavector`. There should be a file named `Bindings.cpp` which calls all
      69             : the `bind_*` functions. The `Bindings.cpp` file is quite simple and should
      70             : `include <boost/python.hpp>`, forward declare the `bind_*` functions, and then
      71             : have `BOOST_PYTHON_MODULE` function. For example,
      72             : 
      73             : \code{.cpp}
      74             : #include <boost/python.hpp>
      75             : 
      76             : namespace py_bindings {
      77             : void bind_datavector();
      78             : }  // namespace py_bindings
      79             : 
      80             : BOOST_PYTHON_MODULE(_DataStructures) {
      81             :   Py_Initialize();
      82             :   py_bindings::bind_datavector();
      83             : }
      84             : \endcode
      85             : 
      86             : Note that the library name is passed to `BOOST_PYTHON_MODULE` and is prefixed
      87             : with an underscore. The underscore is important!
      88             : 
      89             : The `DataVector` bindings serve as an example with code comments on how to write
      90             : bindings for a class. There is also extensive documentation available directly
      91             : from [Boost.Python]
      92             : (https://www.boost.org/doc/libs/1_68_0/libs/python/doc/html/index.html)
      93             : and [this GitHub repository](https://github.com/TNG/boost-python-examples) also
      94             : has many helpful examples. SpECTRE currently aims to support both Python 2.7
      95             : and Python 3 and as such all bindings must support both.
      96             : 
      97             : \note Exceptions should be allowed to propagate through the bindings so that
      98             : error handling via exceptions is possible from python rather than having the
      99             : python interpreter being killed with a call to `abort`.
     100             : 
     101             : ## Testing Python Bindings and Code
     102             : 
     103             : All the python bindings must be tested. SpECTRE uses the
     104             : [unittest](https://docs.python.org/3/library/unittest.html) framework
     105             : provided as part of python. To register a test file with CMake use the
     106             : SpECTRE-provided function `spectre_add_python_test` passing as the first
     107             : argument the test name (e.g. `"Unit.DataStructures.Python.DataVector"`), the
     108             : file as the second argument (e.g. `Test_DataVector.py`), and a semicolon
     109             : separated list of labels as the last (e.g. `"unit;datastructures;python"`).
     110             : All the test cases should be in a single class so that the python unit testing
     111             : framework will run all test functions on a single invocation to avoid startup
     112             : cost.
     113             : 
     114             : Below is an example of registering a python test file
     115             : 
     116             : \code
     117             : spectre_add_python_test(
     118             :   "Unit.DataStructures.Python.DataVector"
     119             :   Test_DataVector.py
     120             :   "unit;datastructures;python")
     121             : \endcode
     122             : 
     123             : ## Using The Bindings
     124             : 
     125             : See \ref spectre_using_python "Using SpECTRE's Python"
     126             : 
     127             : ## Notes:
     128             : 
     129             : - All python libraries are dynamic/shared libraries.
     130             : - Exceptions should be allowed to propagate through the bindings so that
     131             :   error handling via exceptions is possible from python rather than having the
     132             :   python interpreter being killed with a call to `abort`.

      10             : # Testing SpECTRE with Travis CI {#travis_ci}
      11             : 
      12             : SpECTRE uses
      13             : [TravisCI](https://travis-ci.org/sxs-collaboration/spectre) for
      14             : testing the code.  Multiple build jobs (described below) are launched
      15             : each time a pull request is submitted or updated.  In addition, a
      16             : daily cron job is run.  Travis will also launch these build jobs each
      17             : time you push to a branch on your fork of SpECTRE if you enable
      18             : it. (Go to Travis, click on your name in the upper right corner, and
      19             : then towards the bottom, enable testing on your fork of SpECTRE.)
      20             : 
      21             : For pull requests, you can view the Travis CI build by clicking on
      22             : `Details` next to the `Travis build failed` or `Travis build
      23             : succeeded` in the section for `All checks have failed` or `All checks
      24             : have passed` towards the bottom of the pull request.  You can login to
      25             : Travis using your GitHub credentials in order to cancel your builds,
      26             : and you should do so if you update your pull request while it is
      27             : building.
      28             : 
      29             : ## What is tested {#what-is-tested}
      30             : 
      31             : The Travis report lists the build jobs which will each have either a
      32             : green check mark if it passes, a red `X` if it has failed, or a yellow
      33             : binary if the build is in progress.  Clicking on a build job will
      34             : display the log of the build.
      35             : 
      36             : The following build jobs are launched:
      37             : * CHECK_COMMITS runs the script `tools/CheckCommits.sh` and fails the build if
      38             : any casing of the words in the list below is the first word of the commit
      39             : message.  This allows developers to flag their commits with these keywords to
      40             : indicate that a pull request should not be merged in its current state.
      41             :   - fixup
      42             :   - wip (for work in progress)
      43             :   - fixme
      44             :   - deleteme
      45             :   - rebaseme
      46             :   - testing
      47             :   - rebase
      48             : * CHECK_FILES runs the script `tools/CheckFiles.sh` (which also runs the script
      49             : `tools/FileTestDefs.sh`) and fails the build if any of the following checks are
      50             : true:
      51             :   - A `c++` file (i.e. `*.hpp`, `*.cpp`, or `*.tpp` file) contains a
      52             :   line over 80 characters
      53             :   - A file contains a tab character
      54             :   - A file has white space at the end of the line
      55             :   - A file has a carriage return character
      56             :   - A file is missing the license line
      57             :   - A `*.hpp` file is missing `pragma once`
      58             :   - A file does not end with a newline
      59             :   - A `c++` file includes any of the following headers
      60             :     * `<iostream>`  (useless when running in parallel)
      61             :     * `Utilities/TmplDebugging.hpp`  (used only for debugging)
      62             :   - A `c++` file contains a `namespace` ending in `_details` (use `_detail`)
      63             :   - A `c++` file contains a `struct TD` or `class TD` (used only for debugging)
      64             :   - A `c++` file contains `std::enable_if` (use `Requires` instead)
      65             :   - A `c++` file contains additional text after `/*!` (does not render correctly
      66             :   in doxygen)
      67             :   - A `c++` file contains `Ls` (use `List` instead)
      68             :   - A `c++` test uses either:
      69             :     * `TEST_CASE` (use `SPECTRE_TEST_CASE` instead)
      70             :     * `Approx` (use `approx` instead)
      71             : * RUN_CLANG_TIDY runs the script `.travis/RunClangTidy.sh` which runs
      72             : `clang-tidy` on all files which were modified.  This is done for both `Release`
      73             : and `Debug` builds.
      74             : * TEST_CHECK_FILES runs `tools/CheckFiles.sh --test` which tests the checks
      75             : performed in the CHECK_FILES build.
      76             : * The other builds compile the code and run the tests for both
      77             : `Release` and `Debug` builds, for the `gcc` and `clang` compilers
      78             : using a Linux OS, and the `AppleClang` compiler for `OS X`.
      79             : * The `gcc Debug` build will fail if there are `doxygen` warnings.
      80             : 
      81             : ## How to perform the checks locally {#perform-checks-locally}
      82             : 
      83             : Before pushing to GitHub and waiting for Travis to perform the checks it is
      84             : useful to perform at least the following tests locally:
      85             : - **Unit tests:** Perform a `make test-executables` and then execute `ctest` to
      86             :   run all unit tests. As for `make` you can append a `-jN` flag to `ctest` to
      87             :   run in parallel on `N` cores. To run only a subset of the tests you can use
      88             :   one of the other keywords that the tests are labeled with, such as `ctest -L
      89             :   datastructures`. To run only particular tests you can also execute `ctest -R
      90             :   TEST_NAME` instead, where `TEST_NAME` is a regular expression matching the
      91             :   test identifiers such as `Unit.DataStructures.Mesh`. Pass the flag
      92             :   `--output-on-failure` to get output from failed tests. Consult `ctest -h` for
      93             :   further options.
      94             : - **clang-tidy:** In a clang build directory, run `make clang-tidy
      95             :   FILE=SOURCE_FILE` where `SOURCE_FILE` is a relative or absolute path to a
      96             :   `.cpp` file. To perform this check for all source files that changed in your
      97             :   pull request, `make clang-tidy-hash HASH=UPSTREAM_HEAD` where `UPSTREAM_HEAD`
      98             :   is the hash of the commit that your pull request is based on, usually the
      99             :   `HEAD` of the `upstream/develop` branch.
     100             : - **Documentation:** To render the documentation for the current state
     101             :   of the source tree the command `make doc` (or `make doc-check` to
     102             :   highlight warnings) can be used, placing its result in the `docs`
     103             :   directory in the build tree.  Once code has been made into a pull
     104             :   request to GitHub, the documentation can be rendered locally using
     105             :   the `tools/pr-docs` script.  To view the documentation, simply open the
     106             :   `index.html` file in the `html` subdirectory in a browser. Some functionality
     107             :   requires a web server (e.g. citation popovers), so just run a
     108             :   `python3 -m http.server` in the `html` directory to enable this.
     109             : - **IWYU:** We experimented for a time using IWYU (include what you
     110             :   use) as an automated check for correct includes and forward
     111             :   declarations.  Unfortunately it gave many incorrect suggestions.  We
     112             :   have decided to no longer have a IWYU check for Travis, but have
     113             :   left support for IWYU so that it can be used locally.  To do so just
     114             :   for the changed files in a pull request run `make iwyu-hash
     115             :   HASH=UPSTREAM_HEAD`. Since IWYU requires `USE_PCH=OFF` you can
     116             :   create a separate build directory and append `-D USE_PCH=OFF` to the
     117             :   usual `cmake` call. Note that it is very easy to incorrectly install
     118             :   IWYU (if not using the Docker container) and generate nonsense
     119             :   errors.  Note that we have left IWYU pragmas in the code, but no
     120             :   longer require they be added so that IWYU gives no errors when run.
     121             :   As IWYU is still under development, we plan to investigate using it
     122             :   again in the future.
     123             : 
     124             : ## Travis setup {#travis-setup}
     125             : 
     126             : * The `gcc Debug` build runs code coverage for each Travis build.
     127             : 
     128             : ## Troubleshooting {#travis-troubleshooting}
     129             : 
     130             : * Occasionally, a build job will fail because of a problem with Travis
     131             : (e.g. it times out).  Clicking on the circular arrow icon on the far
     132             : right of the build job will restart it.  All build jobs can be
     133             : restarted by clicking on the `Restart build` button to the right of
     134             : the pull request description.
     135             : * Travis caches some things between builds.  Occasionally this may
     136             : cause a problem leading to strange build failures.  If you suspect
     137             : this may be the case, get a SpECTRE owner to click on the `More
     138             : options` button in the top right, choose `Caches` and delete the cache
     139             : for your pull request.
     140             : 
     141             : ## Precompiled Headers and ccache {#precompiled-headers-ccache}
     142             : 
     143             : Getting ccache to work with precompiled headers on TravisCI is a little
     144             : challenging. The header to be precompiled is
     145             : `${SPECTRE_SOURCE_DIR}/tools/SpectrePch.hpp` and is symbolically linked to
     146             : `${SPECTRE_BUILD_DIR}/SpectrePch.hpp`. The configuration that seems to work is
     147             : specifying the environment variables:
     148             : 
     149             : \code{.sh}
     150             : CCACHE_COMPILERCHECK=content
     151             : CCACHE_EXTRAFILES="${SPECTRE_SOURCE_DIR}/tools/SpectrePch.hpp"
     152             : CCACHE_IGNOREHEADERS="${SPECTRE_BUILD_DIR}/SpectrePch.hpp:${SPECTRE_BUILD_DIR}/SpectrePch.hpp.gch"
     153             : \endcode
     154             : 
     155             : On macOS builds we haven't yet had success with using `ccache` with a
     156             : precompiled header. We disable the precompiled header and build in debug mode
     157             : only to have reasonable build times.
     158             : 
     159             : ## Build Stages {#travis-build-stages}
     160             : 
     161             : In order to avoid timeouts we build SpECTRE in various stages, carrying over the
     162             : ccache from one stage to the next. This allows us to avoid recompiling the code
     163             : in the next stage (there is some small overhead from running ccache instead of
     164             : not doing anything at all). The first stage builds all the SpECTRE libraries but
     165             : none of the executables or testing libraries. The second stage builds builds the
     166             : test executables, runs the tests, and also runs ClangTidy, include-what-you-use,
     167             : and various other checks. Another stage could be added that builds some of the
     168             : test libraries if necessary.
     169             : 
     170             : ## Caching Dependencies on macOS Builds {#caching-mac-os}
     171             : 
     172             : On macOS builds we cache all of our dependencies, like LIBXSMM and
     173             : Charm++. These are cached in `$HOME/mac_cache`. Ultimately this saves about
     174             : 10-12 minutes even when compared to using ccache to cache the object files from
     175             : building the dependencies. We also cache `$HOME/Library/Caches/Homebrew`, which
     176             : is where Homebrew keeps the downloaded formulas. By caching the Homebrew bottles
     177             : we are able to avoid brew formulas building from source because a tarball of the
     178             : package was not available at the time.

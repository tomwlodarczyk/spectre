       5             : # SFINAE {#sfinae}
       6             : 
       7             : SFINAE, Substitution Failure Is Not An Error, means that if a deduced template
       8             : substitution fails, compilation must continue. This can be exploited to make
       9             : decisions at compile time. See [here](http://nilsdeppe.com/posts/tmpl-part1)
      10             : for a discussion using `std::enable_if` to remove certain functions from
      11             : overload resolution or certain template specializations from name lookup.
      12             : Another method of controlling name lookup resolution is using `std::void_t`
      13             : which is implemented as `cpp17::void_t` in SpECTRE. `void_t` is a metafunction
      14             : from types to `void`, that is
      15             : 
      16             : ```cpp
      17             : template <typename... Args>
      18             : using void_t = void;
      19             : ```
      20             : 
      21             : `void_t` is useful when used in combination with `decltype` and `std::declval`
      22             : to probe if a type has certain members. For example, we can implement a type
      23             : trait to check if a type `T` is iterable by first have the general definition
      24             : inherit from `std::false_type` as follows,
      25             : 
      26             : ```cpp
      27             : template <typename T, typename = void>
      28             : struct is_iterable : std::false_type {};
      29             : ```
      30             : 
      31             : Next we will have specialization that uses `void_t` to check if the type `T`
      32             : has a `begin()` and `end()` function.
      33             : 
      34             : ```cpp
      35             : template <typename T>
      36             : struct is_iterable<T, cpp17::void_t<decltype(std::declval<T>().begin(),
      37             :                                              std::declval<T>().end())>>
      38             :     : std::true_type {};
      39             : ```
      40             : 
      41             : What is happening here? Well, we use `std::declval` to convert the type `T`
      42             : to a reference type, which allows us to call member functions inside `decltype`
      43             : expressions without having to construct an object. First we try to call the
      44             : member function `begin()` on `std::declval<T>()`, and if that succeeds we
      45             : throw away the result using the comma operator. Next we try to call `end()`
      46             : on `std::declval<T>()`, which, if it succeeds we get the return type of
      47             : using `decltype`. Note that `decltype` is important because we can only call
      48             : member functions on reference types inside of `decltype`, not just anywhere.
      49             : Finally, if all this succeeded use `void_t` to metareturn `void`, otherwise
      50             : the template parameters of `void_t` fail to evaluate and the specialization
      51             : cannot be resolved during name lookup. We could just as well use
      52             : 
      53             : ```cpp
      54             : template <typename T>
      55             : struct is_iterable<T, cpp17::void_t<decltype(std::declval<T>().begin()),
      56             :                                     decltype(std::declval<T>().end())>>
      57             :     : std::true_type {};
      58             : ```
      59             : 
      60             : Which of the two implementations of the `is_iterable` is preferred is simply
      61             : a matter of taste, both behave correctly.
      62             : 
      63             : If you're reading closely you might wonder why the `void_t` is necessary at
      64             : all, why not just `decltype(...)`? Well the reason is that since the default
      65             : template parameter metavalue is `void`, the specialization cannot be resolved
      66             : during name lookup unless the second template parameter in the specialization
      67             : is either `void` as well or is explicitly specified when the class template
      68             : is being invoked. Thus, the clearest implementation probably is
      69             : 
      70             : ```cpp
      71             : template <typename T, typename = cpp17::void_t<>>
      72             : struct is_iterable : std::false_type {};
      73             : 
      74             : template <typename T>
      75             : struct is_iterable<T, cpp17::void_t<decltype(std::declval<T>().begin(),
      76             :                                              std::declval<T>().end())>>
      77             :     : std::true_type {};
      78             : ```
      79             : 
      80             : You could now also define a helper type alias and constexpr boolean
      81             : 
      82             : ```cpp
      83             : template <typename T>
      84             : using is_iterable_t = typename is_iterable<T>::type;
      85             : 
      86             : template <typename T>
      87             : constexpr bool is_iterable_v = is_iterable<T>::value;
      88             : ```

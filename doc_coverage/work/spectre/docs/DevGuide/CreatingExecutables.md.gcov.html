       5             : # Creating Executables {#dev_guide_creating_executables}
       6             : 
       7             : There are several different types of executables that can be built:
       8             : - An executable that uses Charm++ to run in parallel,
       9             :   e.g. `Executables/ParallelInfo`
      10             : - An executable that does not use Charm++, does not run in parallel, and
      11             :   supplies its own `main`, e.g. `Executables/Benchmark`
      12             : - An executable that uses Charm++ to run in parallel but supplies its own `main`
      13             : - An executable that uses custom compilation or linking flags,
      14             :   e.g. `DebugPreprocessor`
      15             : - Executables used for evolutions or elliptic solves
      16             : 
      17             : ### Executable Using Charm++ for Parallelization
      18             : 
      19             : A simple example of an executable using Charm++ for parallelization is in
      20             : `src/Executables/Examples/HelloWorld`. To add an executable as a target, use
      21             : the CMake function `add_spectre_parallel_executable` which takes as its four
      22             : arguments
      23             : - the name of the executable
      24             : - the name of the path relative to src that contains the header file (which is
      25             :   the name of the executable with extension `.hpp`) that will define structs and
      26             :   functions to be described below
      27             : - the name of the metavariables struct described below
      28             : - the list of SpECTRE libraries needed for the executable to link
      29             : 
      30             : For example,
      31             : 
      32             : ```
      33             : set(LIBS_TO_LINK
      34             :   Informer
      35             :   Utilities
      36             :   )
      37             : 
      38             : add_spectre_parallel_executable(
      39             :   SingletonHelloWorld
      40             :   Executables/Examples/HelloWorld
      41             :   Metavars
      42             :   "${LIBS_TO_LINK}"
      43             :   )
      44             : ```
      45             : will create the executable `SingletonHelloWorld` from
      46             : SingletonHelloWorld.hpp which is found in the directory
      47             : `src/Executables/Examples/HelloWorld`.  `Metavars` will be used as the
      48             : metavariables struct, and `SingletonHelloWorld` will link against the
      49             : `%Informer` and `Utilities` SpECTRE libraries.
      50             : 
      51             : The header file from which an executable is generated must define a
      52             : metavariables struct that can be thought of as a compile-time input file that
      53             : defines what the executable will do.  The `Metavars` struct for
      54             : `SingletonHelloWorld` is a minimal example of a metavariables struct.
      55             : 
      56             : \snippet SingletonHelloWorld.hpp executable_example_metavariables
      57             : 
      58             : Each metavariables must define an enum class `Phase` with the phases of the
      59             : executable (which must include `Initialization` and `Exit`), and a static
      60             : function `determine_next_phase`.  In the example, the only additional phase is
      61             : `Execute`, and the phases are executed in order.  Charm++ will execute each
      62             : phase until it detects that nothing is happening (quiescence detection).  As
      63             : this represents a global synchronization point, the number of phases should be
      64             : minimized in order to exploit the power of SpECTRE.
      65             : 
      66             : Each metavariables must define a type alias `component_list` that is a
      67             : `tmpl::list` of the parallel components used by the executable.
      68             : `SingletonHelloWorld` defines a single component `HelloWorld`
      69             : 
      70             : \snippet SingletonHelloWorld.hpp executable_example_singleton
      71             : 
      72             : which specifies via the `chare_type` type alias that it is a singleton parallel
      73             : component which means that only one such object will exist across all processors
      74             : used by the executable.  Each component must define the static functions
      75             : `initialize`, which is executed during the `Initialization` phase, and
      76             : `execute_next_phase` which is executed during the phases (other than
      77             : `Initialization` and `Exit`) defined in the metavariables struct.  In
      78             : `SingletonHelloWorld`, nothing is done during the initialization phase, while
      79             : the `PrintMessage` action is called during the `Execute` phase.
      80             : 
      81             : \snippet  SingletonHelloWorld.hpp executable_example_action
      82             : 
      83             : The `PrintMessage` action is executed on whatever process the singleton
      84             : component is created upon, and prints a message.
      85             : 
      86             : Executables can read in an input file (specified by the `--input-file`
      87             : argument) that will be parsed when the executable begins.  %Options
      88             : specified in the input file can be used to either place items in the
      89             : Parallel::ConstGlobalCache (by specifying tags in the
      90             : `const_global_cache_tags` type alias of the metavariables, component
      91             : and action structs), to construct items in the db::DataBox of
      92             : components during initialization (by specifying tags in the
      93             : `initialization_tags` type alias of action struct), or be passed to
      94             : the `allocate_array` function of an array component (by specifying
      95             : tags in the `allocation_tags` type alias of the component).
      96             : `SingletonHelloWorld` specifies a single option
      97             : 
      98             : \snippet SingletonHelloWorld.hpp executable_example_options
      99             : 
     100             : which a string specifying a name that will be placed into the constant global
     101             : cache.  The string is fetched when performing the `PrintMessage` action. Items
     102             : in the constant global cache are stored once per node that the executable runs
     103             : on. An example input file for `SingletonHelloWorld` can be found in
     104             : `tests/InputFiles/ExampleExecutables/SingletonHelloWorld.yaml` and shows how to
     105             : specify the options (lines beginning with a `#` are comments and can be
     106             : ignored).
     107             : 
     108             : In addition to defining the metavaribles and component structs, the header for
     109             : an executable must define two lists of functions
     110             : 
     111             : \snippet  SingletonHelloWorld.hpp executable_example_charm_init
     112             : 
     113             : that are executed at startup by Charm++ on each node and processor the
     114             : executable runs on.
     115             : 
     116             : Furthermore among the included header files
     117             : 
     118             : \snippet  SingletonHelloWorld.hpp executable_example_includes
     119             : 
     120             : must be the appropriate header for each parallel component type, which in the
     121             : `SingletonHelloWorld` example is `AlgorithmSingleton.hpp`.  Note that
     122             : these headers are not in the source tree, but are generated automatically when
     123             : the code is compiled.
     124             : 
     125             : See [the Parallelization documentation](group__ParallelGroup.html#details)
     126             : for more details.
     127             : 
     128             : ### Executable Using Charm++ with Custom main()
     129             : 
     130             : While this is technically possible, it has not been tested. We recommend using
     131             : the Charm++ supplied main chare mechanism for the time being.
     132             : 
     133             : ### Executable Not Using Charm++
     134             : 
     135             : An example of an executable that does not use Charm++ for parallelization but
     136             : still can use all other infrastructure in SpECTRE is in
     137             : `src/Executables/Examples/HelloWorldNoCharm`. Adding a non-Charm++ executable to
     138             : SpECTRE mostly follows the standard way of adding an executable using CMake. The
     139             : only deviation is that the `CMakeLists.txt` file must tell Charm++ not to add a
     140             : `main()` by passing the link flags `-nomain-module -nomain`. This is done using
     141             : CMake's `set_target_properties`:
     142             : 
     143             : ```
     144             : set_target_properties(
     145             :   ${EXECUTABLE}
     146             :   PROPERTIES LINK_FLAGS "-nomain-module -nomain"
     147             :   )
     148             : ```
     149             : 
     150             : To add the executable as a target you must use the `add_spectre_executable`
     151             : function, which is a light weight wrapper around CMake's `add_executable`.
     152             : For example,
     153             : 
     154             : ```
     155             : add_spectre_executable(
     156             :   ${EXECUTABLE}
     157             :   EXCLUDE_FROM_ALL # Exclude from calls to `make` without a specified target
     158             :   HelloWorld.cpp
     159             :   )
     160             : ```
     161             : 
     162             : You can link in any of the SpECTRE libraries by adding them to the
     163             : `target_link_libraries`, for example:
     164             : 
     165             : ```
     166             : target_link_libraries(
     167             :   ${EXECUTABLE}
     168             :   DataStructures
     169             :   )
     170             : ```
     171             : 
     172             : We recommend that you add a test that the executable properly runs by adding an
     173             : input file to `tests/InputFiles` in an appropriate subdirectory. See
     174             : [`tests/InputFiles/ExampleExecutables/HelloWorldNoCharm.yaml`]
     175             : (https://github.com/sxs-collaboration/spectre/tree/develop/tests/InputFiles/
     176             : ExampleExecutables/HelloWorldNoCharm.yaml)
     177             : for an example.
     178             : The input file is passed to the executable using `--input-file
     179             : path/to/Input.yaml`. In the case of the executable not taking any input file
     180             : this is just used to generate a test that runs the executable.
     181             : 
     182             : For these types of executables `main` can take the usual `(int argc, char
     183             : *argv[])` and parse command line options. Executables not using Charm++ are just
     184             : standard executables that can link in any of the libraries in SpECTRE.
     185             : 
     186             : \warning
     187             : Currently calling `Parallel::abort` results in a segfault deep inside Charm++
     188             : code. However, the error messages from `ASSERT` and `ERROR` are still printed.
     189             : 
     190             : ### Executable With Custom Compilation or Linking Flags
     191             : 
     192             : Use the CMake function `set_target_properties` to add flags to an executable. To
     193             : call a completely custom compiler invocation you should use the
     194             : `add_custom_target` CMake function. The need for the `custom_target` level of
     195             : control is rare and should generally be avoided since it adds quite a bit of
     196             : technical debt to the code base. Thus, it is not explained here. If you are
     197             : certain you need it you can see the `DebugPreprocessor` executable's
     198             : `CMakeLists.txt` file for an example.
     199             : 
     200             : ### Executable Used for Evolution or Elliptic Solve
     201             : 
     202             : Once they are written, see the tutorials specific to evolution and elliptic
     203             : solves.

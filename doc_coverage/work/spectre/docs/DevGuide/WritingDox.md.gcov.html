       9             : # Tutorials, Instructions, and Dev Guide {#writing_dox_writing_help}
      10             : 
      11             : All non-code documentation such as tutorials, installation instructions, and
      12             : the developer guide is written inside Markdown files such as this one. These
      13             : files must be placed inside `docs/Tutorials`, `docs/MainSite`, or
      14             : `docs/DevGuide` according to what they document. Each Markdown file must start
      15             : with the following line
      16             : 
      17             : ``` markdown
      18             : # The title {#the_tag}
      19             : ```
      20             : 
      21             : where `The title` is replaced with your desired title, and `the_tag` with the
      22             : tag you want to use to reference the Markdown file and documentation. Each
      23             : main heading of the file starts with a single octothorpe
      24             : and can have a tag. For example,
      25             : 
      26             : ``` markdown
      27             : # My Section {#file_name_my_section}
      28             : ```
      29             : 
      30             : While the `file_name` portion is not necessary, it is useful for reducing the
      31             : likelihood of reference collisions. You can add a table of contents using the
      32             : Doxygen command <code>\\tableofcontents</code>. All sections, subsections,
      33             : subsubsections, etc. are shown in the table of contents if they have a tag,
      34             : if not they do not appear.
      35             : 
      36             : # C++ Documentation {#writing_dox_cpp_dox_help}
      37             : 
      38             : We require you to add Doxygen documentation for any of the following
      39             : you may have added to the public interface in any hpp file:
      40             : 
      41             : + classes
      42             : + structs
      43             : + enums
      44             : + functions
      45             : + namespaces
      46             : + macros
      47             : + type aliases
      48             : 
      49             : Documentation begins on the line immediately above the declaration with
      50             : either a triple slash `///` or a `/*!`.
      51             : 
      52             : Examples:
      53             : \verbatim
      54             : /// \brief A brief description of the object to be documented
      55             : ///
      56             : /// Doxygen comments can be made
      57             : /// using triple slashes...
      58             : class ExampleClass{
      59             :   ... rest of code
      60             : }
      61             : \endverbatim
      62             : 
      63             : \verbatim
      64             : /*!
      65             :  * \brief A brief description of the object to be documented
      66             :  *
      67             :  * Doxygen comments can also be made
      68             :  * using the "slash-star, star-slash" pattern, in this way.
      69             :  */
      70             : class ExampleClass{
      71             :   ... rest of code
      72             : }
      73             : \endverbatim
      74             : 
      75             : Build your documentation by running `make doc` in your build directory.
      76             : You can then view it by opening `BUILD_DIR/docs/html/index.html` in a
      77             : browser and using the file navigator in Doxygen to locate your file.
      78             : 
      79             : In addition to providing a file directory of all the source files in SpECTRE,
      80             : Doxygen also conveniently provides two additional organizations of the files,
      81             : Topics and Namespaces. To ensure that your documentation is easily found from
      82             : within Doxygen, we recommend that you add any new objects to Topics and any
      83             : new namespaces to Namespaces.
      84             : 
      85             : ## Add your object to an existing Topic:
      86             : 
      87             : Within a Doxygen comment, use the Doxygen keyword
      88             : \verbatim
      89             : \ingroup
      90             : \endverbatim
      91             : 
      92             : followed by the name of the Topic (you can find the list of existing Topics in
      93             : `docs/GroupDefs.hpp`).
      94             : 
      95             : ## Add a new Topic:
      96             : 
      97             : Within `docs/GroupDefs.hpp`, add the name of your Topic (which follows the
      98             : naming convention "YourNameForTopic" followed by "Group") among the rest,
      99             : taking care to keep the list of Topics in alphabetical order.
     100             :

     102             : 
     103             : In the hpp file in which the namespace is declared for the first time,
     104             : add a Doxygen comment to the line directly above the namespace. Subsequent
     105             : files which use this namespace will not need a Doxygen comment.

     109             : ## Put any mathematical expressions in your documentation into LaTeX form:
     110             : 
     111             : Within a Doxygen comment, begin and conclude your expression with
     112             : 
     113             : \verbatim
     114             : \f$
     115             : \endverbatim
     116             : 
     117             : Example:
     118             : \verbatim
     119             : \\\ We define \f$ \xi : = \eta^2 \f$
     120             : \endverbatim
     121             : 
     122             : Note that if this expression spans more than one line,
     123             : your Doxygen comment must of the "slash-star, star-slash" form shown in the
     124             : previous section. One can also use (within a Doxygen comment) the form
     125             : 
     126             : \verbatim
     127             : \f[ expression \f]
     128             : \endverbatim
     129             : 
     130             : to put the expression on its own line. We also encourage you to use the latex
     131             : env `align` for formatting these multiple-line equations.

     133             : ## Cite publications in your documentation {#writing_dox_citations}
     134             : 
     135             : When you refer to publications or books in the documentation, add a
     136             : corresponding entry to `docs/References.bib`. Follow these guidelines when
     137             : editing `docs/References.bib`:
     138             : 
     139             : - Ideally, find the publication on [INSPIRE HEP](https://inspirehep.net), copy
     140             : its BibTeX entry, remove the colon `:` from its key and add the `url` field (see
     141             : below). We remove the colon because it can create problems in HTML related to
     142             : its function as a CSS selector.
     143             : - For publications that are not listed on [INSPIRE HEP](https://inspirehep.net),
     144             : make sure to format the new entry's key in the same style, i.e.
     145             : `(<Author>[a-zA-Z]+)(<Year>[0-9]{4})(<ID>[a-z]*)`. Good keys are, for
     146             : instance, `Einstein1915` or `LVC2016a`. For books you may omit the year.
     147             : - Sort the list of BibTeX entries in the file alphabetically by their keys.
     148             : - Provide open access or preprint information whenever possible. For
     149             : publications available on [arXiv](https://arxiv.org), for instance, add the
     150             : following fields (filling in the correct values):
     151             : ```
     152             : archivePrefix = {arXiv},
     153             : eprint = {1609.00098},
     154             : primaryClass = {astro-ph.HE},
     155             : ```
     156             : - Provide the `url` field whenever possible. If a DOI has been issued by the
     157             : publisher, use `https://doi.org/<doi>` and also fill the `doi` field of the
     158             : entry. Else, use the URL provided by the publisher.
     159             : - Make sure to wrap strings in the BibTeX entries in `{}` when capitalization is
     160             : important, or when BibTeX keywords should be ignored (e.g. `and` in author
     161             : lists).
     162             : 
     163             : To cite an entry from the `docs/References.bib` file in the documentation, use
     164             : the Doxygen keyword
     165             : \verbatim
     166             : \cite
     167             : \endverbatim
     168             : followed by the BibTeX key at the place in the documentation where you want the
     169             : citation to appear. It will render as a numbered link to the bibliography
     170             : page. It will also show a popover when hovering over the link, which displays
     171             : the bibliographic information and provides quick access to the publication.
     172             : 
     173             : ## Include any pictures that aid in the understanding of your documentation:
     174             : 
     175             : First, please compress your image to be under 130kB. As most images will be
     176             : diagrammatical in nature this is certainly achievable!
     177             : 
     178             : Second, add the directory that will contain your image to the `docs/Images/`
     179             : directory. Create the appropriate directories such that the directory structure
     180             : in `docs/Images` matches that of `src` and `tests/Unit`.
     181             : 
     182             : Finally, include the image by using the Doxygen keyword
     183             : 
     184             : \verbatim
     185             : \image html NameOfImage.png
     186             : \endverbatim
     187             : 
     188             : in your hpp file.

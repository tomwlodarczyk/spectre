       5             : # Writing Unit Tests {#writing_unit_tests}
       6             : 
       7             : Unit tests are placed in the appropriate subdirectory of `tests/Unit`, which
       8             : mirrors the directory hierarchy of `src`. The tests are all compiled into
       9             : individual libraries to keep link time of testing executables low. Typically
      10             : there should be one test library for each production code library. For example,
      11             : we have a `DataStructures` library and a `Test_DataStructures` library. When
      12             : adding a new test there are several scenarios that can occur, which are outlined
      13             : below.
      14             : 
      15             : - You are adding a new source file to an existing test library:<br>
      16             :   If you are adding a new source file in a directory that already has a
      17             :   `CMakeLists.txt` simply create the source file, which should be named
      18             :   `Test_ProductionCodeFileBeingTest.cpp` and add that to the `LIBRARY_SOURCES`
      19             :   in the `CMakeLists.txt` file in the same directory you are adding the `cpp`
      20             :   file.<br>
      21             :   If you are adding a new source file to a library but want to place it in a
      22             :   subdirectory you must first create the subdirectory. To provide a concrete
      23             :   example, say you are adding the directory `TensorEagerMath` to
      24             :   `tests/Unit/DataStructures`. After creating the directory you must add a call
      25             :   to `add_subdirectory(TensorEagerMath)` to
      26             :   `tests/Unit/DataStructures/CMakeLists.txt` *before* the call to
      27             :   `add_test_library` and *after* the `LIBRARY_SOURCES` are set. Next add the
      28             :   file `tests/Unit/DataStructures/TensorEagerMath/CMakeLists.txt`, which should
      29             :   add the new source files by calling `set`, e.g.
      30             :   ```
      31             :   set(LIBRARY_SOURCES
      32             :       ${LIBRARY_SOURCES}
      33             :       Test_ProductionCodeFileBeingTest.cpp
      34             :       PARENT_SCOPE)
      35             :   ```
      36             :   The `PARENT_SCOPE` flag tells CMake to make the changes visible in the
      37             :   CMakeLists.txt file that called `add_subdirectory`. You can now add the
      38             :   `Test_ProductionCodeFileBeingTested.cpp` source file.
      39             : - You are adding a new directory:<br>
      40             :   If the directory is a new lowest level directory you must add a
      41             :   `add_subdirectory` call to `tests/Unit/CMakeLists.txt`. If it is a new
      42             :   subdirectory you must add a `add_subdirectory` call to the
      43             :   `CMakeLists.txt` file in the directory where you are adding the
      44             :   subdirectory. Next you should read the part on adding a new test library.
      45             : - You are adding a new test library:<br>
      46             :   After creating the subdirectory for the new test library you must add a
      47             :   `CMakeLists.txt` file. See `tests/Unit/DataStructures/CMakeLists.txt` for
      48             :   an example of one. The `LIBRARY` and `LIBRARY_SOURCES` variables set the name
      49             :   of the test library and the source files to be compiled into it. The library
      50             :   name should be of the format `Test_ProductionLibraryName`, for example
      51             :   `Test_DataStructures`. The library sources should be only the source files in
      52             :   the current directory. The `add_subdirectory` command can be used to add
      53             :   source files in subdirectories to the same library as is done in
      54             :   `tests/Unit/CMakeLists.txt`. The `CMakeLists.txt` in
      55             :   `tests/Unit/DataStructures/TensorEagerMath` is an example of how to add source
      56             :   files to a library from a subdirectory of the library. Note that the setting
      57             :   of `LIBRARY_SOURCES` here first includes the current `LIBRARY_SOURCES` and at
      58             :   the end specifies `PARENT_SCOPE`. The `PARENT_SCOPE` flag tells CMake to
      59             :   modify the variable in a scope that is visible to the parent directory,
      60             :   i.e. the `CMakeLists.txt` that called `add_subdirectory`.<br>
      61             :   Finally, in the `CMakeLists.txt` of your new library you must call
      62             :   `add_test_library`. Again, see `tests/Unit/DataStructures/CMakeLists.txt` for
      63             :   an example. The `add_test_library` function adds a test library with the name
      64             :   of the first argument and the source files of the third argument. The second
      65             :   argument is the path of the library's directory relative to `tests/Unit`. For
      66             :   example, for `Test_DataStructures` it is simply `DataStructures`. The fourth
      67             :   and final argument to `add_test_library` are the libraries that must be
      68             :   linked. Typically this should only be the production library you're
      69             :   testing. For example, `Test_DataStructures` should specify only
      70             :   `DataStructures` as the library to link. If you are testing a header-only
      71             :   "library" then you do not link any libraries (they must be linked in by the
      72             :   libraries actually testing your dependencies). In this case the last argument
      73             :   should be `"" # Header-only, link dependencies included from testing lib`
      74             : 
      75             : All tests must start with
      76             : ```cpp
      77             : // Distributed under the MIT License.
      78             : // See LICENSE.txt for details.
      79             : 
      80             : #include "tests/Unit/TestingFramework.hpp"
      81             : ```
      82             : The file `tests/Unit/TestingFramework.hpp` must always be the first include in
      83             : the test file and must be separated from the STL includes by a blank line. All
      84             : classes and free functions should be in an anonymous/unnamed namespace, e.g.
      85             : ```cpp
      86             : namespace {
      87             : class MyFreeClass {
      88             :   /* ... */
      89             : };
      90             : 
      91             : void my_free_function() noexcept {
      92             :   /* ... */
      93             : }
      94             : }  // namespace
      95             : ```
      96             : This is necessary to avoid symbol redefinition errors during linking.
      97             : 
      98             : Test cases are added by using the `SPECTRE_TEST_CASE` macro. The first argument
      99             : to the macro is the test name, e.g. `"Unit.DataStructures.Tensor"`, and the
     100             : second argument is a list of tags. The tags list is a string where each element
     101             : is in square brackets. For example, `"[Unit][DataStructures]"`. The tags should
     102             : only be the type of test, in this case `Unit`, and the library being tested, in
     103             : this case `DataStructures`. The `SPECTRE_TEST_CASE` macro should be treated as a
     104             : function, which means that it should be followed by `{ /* test code */ }`. For
     105             : example,
     106             : \snippet Test_Tensor.cpp example_spectre_test_case
     107             : From within a `SPECTRE_TEST_CASE` you are able to do all the things you would
     108             : normally do in a C++ function, including calling other functions, setting
     109             : variables, using lambdas, etc.
     110             : 
     111             : The `CHECK` macro in the above example is provided by
     112             : [Catch2](https://github.com/catchorg/Catch2) and is used to check conditions. We
     113             : also provide the `CHECK_ITERABLE_APPROX` macro which checks if two `double`s or
     114             : two iterable containers of `double`s are approximately
     115             : equal. `CHECK_ITERABLE_APPROX` is especially useful for comparing `Tensor`s,
     116             : `DataVector`s, and `Tensor<DataVector>`s since it will iterate over nested
     117             : containers as well.
     118             : 
     119             : \warning Catch's `CHECK` statement only prints numbers out to approximately 10
     120             : digits at most, so you should generally prefer `CHECK_ITERABLE_APPROX` for
     121             : checking double precision numbers, unless you want to check that two numbers are
     122             : bitwise identical.
     123             : 
     124             : All unit tests must finish within a few seconds, the hard limit is 5, but having
     125             : unit tests that long is strongly discouraged. They should typically complete in
     126             : less than half a second. Tests that are longer are often no longer testing a
     127             : small enough unit of code and should either be split into several unit tests or
     128             : moved to an integration test.
     129             : 
     130             : #### Discovering New and Renamed Tests
     131             : 
     132             : When you add a new test to a source file or rename an existing test the change
     133             : needs to be discovered by the testing infrastructure. This is done by building
     134             : the target `rebuild_cache`, e.g. by running `make rebuild_cache`.
     135             : 
     136             : #### Testing Pointwise Functions
     137             : 
     138             : Pointwise functions should generally be tested in two different ways. The first
     139             : is by taking input from an analytic solution and checking that the computed
     140             : result is correct. The second is to use the random number generation comparison
     141             : with Python infrastructure. In this approach the C++ function being tested is
     142             : re-implemented in Python and the results are compared. Please follow these
     143             : guidelines:
     144             : 
     145             : - The Python implementation should be in a file with the same name as the source
     146             :   file that is being re-implemented and placed in the same directory as its
     147             :   corresponding `Test_*.cpp` source file.
     148             : - The functions should have the same names as the C++ functions they
     149             :   re-implement.
     150             : - If a function does sums over tensor indices then
     151             :   [`numpy.einsum`](https://docs.scipy.org/doc/numpy/reference/generated/numpy.einsum.html)
     152             :   should be used in Python to provide an alternative implementation of the loop
     153             :   structure.
     154             : - You can import Python functions from other re-implementations in the
     155             :   `tests/Unit/` directory to reduce code duplication. Note that the path you
     156             :   pass to `pypp::SetupLocalPythonEnvironment` determines the directory from
     157             :   which you can import Python modules. Either import modules directly from the
     158             :   `tests/Unit/` directory (e.g. `import
     159             :   PointwiseFunction.GeneralRelativity.Christoffel as christoffel`) or use
     160             :   relative imports like `from . import Christoffel as christoffel`. Don't assume
     161             :   the Python environment is set up in a subdirectory of `tests/Unit/`.
     162             : 
     163             : It is possible to test C++ functions that return by value and ones that return
     164             : by `gsl::not_null`. In the latter case, since it is possible to return multiple
     165             : values, one Python function taking all non-`gsl::not_null` arguments must be
     166             : supplied for each `gsl::not_null` argument to the C++. To perform the test the
     167             : `pypp::check_with_random_values()` function must be called. For example, the
     168             : following checks various C++ functions by calling into `pypp`:
     169             : 
     170             : \snippet Test_Pypp.cpp cxx_two_not_null
     171             : 
     172             : The corresponding Python functions are:
     173             : 
     174             : \snippet PyppPyTests.py python_two_not_null
     175             : 
     176             : #### Testing Failure Cases
     177             : 
     178             : Adding the "attribute" `// [[OutputRegex, Regular expression to match]]`
     179             : before the `SPECTRE_TEST_CASE` macro will force ctest to only pass the
     180             : particular test if the regular expression is found. This can be used to test
     181             : error handling. When testing `ASSERT`s you must mark the `SPECTRE_TEST_CASE` as
     182             : `[[noreturn]]`,
     183             : add the macro `ASSERTION_TEST();` to the beginning of the test, and also have
     184             : the test call `ERROR("Failed to trigger ASSERT in an assertion test");` at the
     185             : end of the test body.
     186             : For example,
     187             : 
     188             : ```cpp
     189             : // [[OutputRegex, Must copy into same size]]
     190             : [[noreturn]] SPECTRE_TEST_CASE("Unit.DataStructures.DataVector.ref_diff_size",
     191             :                                "[DataStructures][Unit]") {
     192             :   ASSERTION_TEST();
     193             : #ifdef SPECTRE_DEBUG
     194             :   DataVector data{1.43, 2.83, 3.94, 7.85};
     195             :   DataVector data_ref;
     196             :   data_ref.set_data_ref(data);
     197             :   DataVector data2{1.43, 2.83, 3.94};
     198             :   data_ref = data2;
     199             :   ERROR("Failed to trigger ASSERT in an assertion test");
     200             : #endif
     201             : }
     202             : ```
     203             : If the `ifdef SPECTRE_DEBUG` is omitted then compilers will correctly flag
     204             : the code as being unreachable which results in warnings.
     205             : 
     206             : You can also test `ERROR`s inside your code. These tests need to have the
     207             : `OutputRegex`, and also call `ERROR_TEST();` at the beginning. The do not need
     208             : the `ifdef SPECTRE_DEBUG` block, they can just call have the code that triggers
     209             : an `ERROR`. For example,
     210             : 
     211             : \snippet Test_AbortWithErrorMessage.cpp error_test_example
     212             : 
     213             : ### Building and Running A Single Test File
     214             : 
     215             : In cases where low-level header files are frequently being altered and the
     216             : changes need to be tested, building `RunTests` becomes extremely time
     217             : consuming. The `RunSingleTest` executable in `tests/Unit/RunSingleTest` allows
     218             : one to compile only a select few of the test source files and only link in the
     219             : necessary libraries. To set which test file and libraries are linked into
     220             : `RunSingleTest` edit the `tests/Unit/RunSingleTest/CMakeLists.txt`
     221             : file. However, do not commit your changes to that file since it is meant to
     222             : serve as an example. To compile `RunSingleTest` use `make RunSingleTest`, and to
     223             : run it use `BUILD_DIR/bin/RunSingleTest Unit.Test.Name`.
     224             : 
     225             : \warning
     226             : `Parallel::abort` does not work correctly in the `RunSingleTest` executable
     227             : because a segfault occurs inside Charm++ code after the abort message is
     228             : printed.

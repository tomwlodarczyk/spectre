      59             : ## Using Docker to obtain a SpECTRE environment
      60             : 
      61             : A [Docker](https://www.docker.com/) image is available from
      62             : [DockerHub](https://hub.docker.com/r/sxscollaboration/spectrebuildenv/) and can
      63             : be used to build SpECTRE on a personal machine.
      64             : 
      65             : **Note**: The Docker image is the recommended way of using SpECTRE on a personal
      66             : Linux machine. Because of the wide variety of operating systems available today
      67             : it is not possible for us to support all configurations. However, using Spack
      68             : as outlined below is a supported alternative to Docker images.
      69             : 
      70             : **NOTE**: If you have SELinux active
      71             : on your system you must figure out how to enable sharing files with the host
      72             : OS. If you receive errors that you do not have permission to access a shared
      73             : directory it is likely that your system has SELinux enabled. One option is to
      74             : disable SELinux at the expense of reducing the security of your system.
      75             : 
      76             : To build with the docker image:
      77             : 
      78             : 1. Clone SpECTRE into SPECTRE_ROOT, a directory of your choice.
      79             : 2. Retrieve the docker image (you may need `sudo` in front of this command)
      80             :    ```
      81             :    docker pull sxscollaboration/spectrebuildenv:latest
      82             :    ```
      83             : 3. Start the docker container (you may need `sudo`)
      84             :    ```
      85             :    docker run -v SPECTRE_ROOT:SPECTRE_ROOT --name CONTAINER_NAME \
      86             :               -i -t sxscollaboration/spectrebuildenv:latest /bin/bash
      87             :    ```
      88             :    (The `--name CONTAINER_NAME` is optional, where CONTAINER_NAME is a name
      89             :    of your choice. If you don't name your container, docker will generate an
      90             :    arbitrary name.)
      91             :    You will end up in a shell in a docker container,
      92             :    as root (you need to be root).
      93             :    Within the container, SPECTRE_ROOT is available and
      94             :    CHARM_DIR is /work/charm. For the following steps, stay inside the docker
      95             :    container as root.
      96             : 4. Make a build directory somewhere inside the container, e.g.
      97             :    /work/spectre-build-gcc, and cd into it.
      98             : 5. Build SpECTRE with
      99             : ```
     100             :    cmake -D CMAKE_Fortran_COMPILER=gfortran-8 \
     101             :          -D CHARM_ROOT=/work/charm/multicore-linux64-gcc SPECTRE_ROOT
     102             : ```
     103             :    then
     104             :    `make -jN`
     105             :    to compile the code, `make test-executables -jN` to compile the test
     106             :    executables, and `ctest` to run the tests.
     107             : 
     108             : Notes:
     109             :   * Everything in your build directory is owned by root, and is
     110             :     accessible only within the container.
     111             :   * You should edit source files in SPECTRE_ROOT in a separate terminal
     112             :     outside the container, and use the container only for compiling and
     113             :     running the code.
     114             :   * If you exit the container (e.g. ctrl-d),
     115             :     your compilation directories are still saved, as is the patch
     116             :     that you have applied to /work/charm and any other changes to
     117             :     the container that you have made.
     118             :     To restart the container, try the following commands
     119             :     (you may need `sudo`):
     120             :     1. `docker ps -a`,
     121             :       to list all containers with their CONTAINER_IDs and CONTAINER_NAMEs,
     122             :     2. `docker start -i CONTAINER_NAME` or `docker start -i CONTAINER_ID`,
     123             :       to restart your container.
     124             :   * You can run more than one shell in the same container, for instance
     125             :     one shell for compiling with gcc and another for compiling
     126             :     with clang.
     127             :     To add a new shell, run `docker exec -it CONTAINER_NAME /bin/bash`
     128             :     (or `docker exec -it CONTAINER_ID /bin/bash`) from
     129             :     a terminal outside the container.
     130             :   * In step 3 above, the `-v SPECTRE_ROOT:SPECTRE_ROOT` maps the directory
     131             :     SPECTRE_ROOT outside the container to SPECTRE_ROOT inside the container.
     132             :     Technically docker allows you to say `-v SPECTRE_ROOT:/my/new/path`
     133             :     to map SPECTRE_ROOT outside the container to any path you want inside
     134             :     the container, but **do not do this**.  Compiling inside the container
     135             :     sets up git hooks in SPECTRE_ROOT that
     136             :     contain hardcoded pathnames to SPECTRE_ROOT *as seen from
     137             :     inside the container*. So if your source paths inside and outside the
     138             :     container are different, commands like `git commit` run *from
     139             :     outside the container* will die with `No such file or directory`.
     140             :   * To compile with clang, the cmake command is
     141             : ```
     142             :     cmake -D CMAKE_CXX_COMPILER=clang++ \
     143             :           -D CMAKE_C_COMPILER=clang \
     144             :           -D CMAKE_Fortran_COMPILER=gfortran-8 \
     145             :           -D CHARM_ROOT=/work/charm/multicore-linux64-clang SPECTRE_ROOT
     146             : ```

     148             : ## Using Singularity to obtain a SpECTRE environment
     149             : 
     150             : [Singularity](http://singularity.lbl.gov/index.html) is a container alternative
     151             : to Docker with better security and nicer integration.
     152             : 
     153             : To use Singularity you must:
     154             : 
     155             : 1. Build [Singularity](http://singularity.lbl.gov/index.html) and add it to your
     156             :    `$PATH`
     157             : 2. `cd` to the directory where you want to store the SpECTRE Singularity image,
     158             :    source, and build directories, let's call it WORKDIR. The WORKDIR must be
     159             :    somewhere in your home directory. If this does not work for you, follow the
     160             :    Singularity instructions on setting up additional [bind
     161             :    points](http://singularity.lbl.gov/docs-mount). Once inside the WORKDIR,
     162             :    clone SpECTRE into `WORKDIR/SPECTRE_ROOT`.
     163             : 3. Run `singularity build spectre.img
     164             :    docker://sxscollaboration/spectrebuildenv:latest`.
     165             : 4. To start the container run `singularity shell spectre.img` and you
     166             :    will be dropped into a bash shell.
     167             : 5. Run `cd SPECTRE_HOME && mkdir build && cd build` to set up a build
     168             :    directory.
     169             : 6. To build SpECTRE run
     170             : ```
     171             :    cmake -D CMAKE_Fortran_COMPILER=gfortran-8 \
     172             :          -D CHARM_ROOT=/work/charm/multicore-linux64-gcc SPECTRE_ROOT
     173             : ```
     174             :    followed by
     175             :    `make -jN` where `N` is the number of cores to build on in parallel.
     176             : 
     177             : Notes:
     178             : - You should edit source files in SPECTRE_ROOT in a separate terminal outside
     179             :   the container, and use the container only for compiling and running the code.
     180             : - Unlike Docker, Singularity does not keep the state between runs. However, it
     181             :   shares the home directory with the host OS so you should do all your work
     182             :   somewhere in your home directory.
     183             : - To run more than one container just do `singularity shell spectre.img` in
     184             :   another terminal.
     185             : - Since the data you modify lives on the host OS there is no need to worry about
     186             :   losing any data, needing to clean up old containers, or sharing data between
     187             :   containers and the host.
     188             : - To build with clang, the CMake command is:
     189             : ```
     190             :     cmake -D CMAKE_CXX_COMPILER=clang++ \
     191             :           -D CMAKE_C_COMPILER=clang \
     192             :           -D CMAKE_Fortran_COMPILER=gfortran-8 \
     193             :           -D CHARM_ROOT=/work/charm/multicore-linux64-clang SPECTRE_ROOT
     194             : ```

     196             : ## Using Spack to set up a SpECTRE environment
     197             : 
     198             : SpECTRE's dependencies can be installed with
     199             : [Spack](https://github.com/LLNL/spack), a package manager tailored for HPC use.
     200             : Install Spack by cloning it into `SPACK_DIR` (a directory of your choice),
     201             : then add `SPACK_DIR/bin` to your `PATH`.
     202             : 
     203             : For security, it is good practice to make Spack use the system's OpenSSL
     204             : rather than allow it to install a new copy — see Spack's documentation for
     205             : [instructions](https://spack.readthedocs.io/en/latest/getting_started.html#openssl).
     206             : You may need to install the development version of OpenSSL:
     207             : * On Ubuntu (16.04), run `sudo apt-get install libssl-dev`
     208             : * On Fedora (27), run `sudo dnf install openssl-devel`
     209             : 
     210             : Spack works well with a module environment. We recommend
     211             : [LMod](https://github.com/TACC/Lmod), which is available on many systems:
     212             : * On macOS, install LMod from [brew](https://brew.sh/), then source the LMod
     213             :   shell script by adding `. /usr/local/Cellar/lmod/YOUR_VERSION_NUMBER/init/sh`
     214             :   to your `.bash_profile`.
     215             : * On Ubuntu, run `sudo apt-get install -y lmod` and, for Ubuntu 17.04, add
     216             :   `. /etc/profile.d/lmod.sh` to your `.bashrc`. For Ubuntu 16.04, the correct
     217             :   path to add is `. /usr/share/lmod/lmod/init/bash`.
     218             : * On Arch Linux, run `yaourt -Sy lmod` and add `. /etc/profile.d/lmod.sh` to
     219             :   your `.bashrc`,
     220             : * On Fedora/RHEL, GNU Environment Modules comes out-of-the-box and works equally
     221             :   well.
     222             : * Instructions for other Linux distros are available online.
     223             : 
     224             : To use modules with Spack, enable Spack's shell support by adding
     225             : `. SPACK_DIR/share/spack/setup-env.sh` to your `.bash_profile` or `.bashrc`.
     226             : 
     227             : Once you have Spack installed and configured with OpenSSL and LMod, you can
     228             : install the SpECTRE dependencies using
     229             : ```
     230             : spack install blaze
     231             : spack install brigand@master
     232             : spack install libsharp -openmp -mpi
     233             : spack install catch
     234             : spack install gsl
     235             : spack install jemalloc # or from your package manager
     236             : spack install libxsmm
     237             : spack install yaml-cpp@develop
     238             : ```
     239             : You can also install CMake, OpenBLAS, Boost, and HDF5 from Spack.
     240             : To load the packages you've installed from Spack run `spack load PACKAGE`,
     241             : or (equivalently) use the `module load` command.
     242             : 
     243             : **Note**: Spack allows very flexible configurations and
     244             : it is recommended you read the [documentation](https://spack.readthedocs.io) if
     245             : you require features such as packages installed with different compilers.
     246             : 
     247             : **Note**: On a Mac, you may need to `spack install
     248             : yaml-cpp@develop~shared` (note that is a tilde and not a dash in front
     249             : of shared) in order to force the building of the static libraries in
     250             : order to avoid dynamic linking errors.
     251             : 
     252             : ### Building SpECTRE
     253             : 
     254             : After the dependencies have been installed, Charm++ and SpECTRE can be compiled.
     255             : Follow these steps:
     256             : 
     257             : 1.  Clone [SpECTRE](https://github.com/sxs-collaboration/spectre) into
     258             :     `SPECTRE_ROOT`, a directory of your choice.
     259             : 2.  Install Charm++:
     260             :   * Clone [Charm++](http://charm.cs.illinois.edu/software) into `CHARM_DIR`,
     261             :     again a directory of your choice.
     262             :   * In `CHARM_DIR`, run
     263             :     `git checkout v6.8.0` to switch to a supported, stable release of Charm++.
     264             :   * Charm++ is compiled by running
     265             :     `./build charm++ ARCH OPTIONS`.
     266             :     To figure out the correct target architecture and options, you can simply
     267             :     run `./build`; the script will then ask you questions to guide you towards
     268             :     the correct settings (see notes below for additional details).
     269             :     Then compile Charm++.
     270             :     The Charm++ build will be located in a new directory,
     271             :     `CHARM_DIR/ARCH_OPTS`, whose name may (or may not) have some of the options
     272             :     appended to the architecture.
     273             :   * The SpECTRE repo contains a patch that must be applied to Charm++ *after*
     274             :     Charm++ has been compiled. While still in `CHARM_DIR`, apply this patch by
     275             :     running `git apply SPECTRE_ROOT/support/Charm/v6.8.patch`.
     276             :   * On macOS 10.12 it is necessary to patch the STL implementation. Insert
     277             :     \code
     278             :     #ifndef _MACH_PORT_T
     279             :     #define _MACH_PORT_T
     280             :     #include <sys/_types.h> /* __darwin_mach_port_t */
     281             :     typedef __darwin_mach_port_t mach_port_t;
     282             :     #include <pthread.h>
     283             :     mach_port_t pthread_mach_thread_np(pthread_t);
     284             :     #endif /* _MACH_PORT_T */
     285             :     \endcode
     286             :     into the front of
     287             :     `/Library/Developer/CommandLineTools/usr/include/c++/v1/__threading_support`
     288             : 3.  Return to `SPECTRE_ROOT`, and create a build dir by running
     289             :     `mkdir build && cd build`
     290             : 4.  Build SpECTRE with
     291             :     `cmake -D CHARM_ROOT=CHARM_DIR/ARCH_OPTS SPECTRE_ROOT`
     292             :     then
     293             :     `make -jN`
     294             :     to compile the code.
     295             : 5.  Run the tests with
     296             :     `make test-executables && ctest`.
     297             : 
     298             : **Notes**:
     299             : * For more details on building Charm++, see the directions
     300             :   [here](http://charm.cs.illinois.edu/manuals/html/charm++/A.html)
     301             :   The correct target is `charm++` and, for a personal machine, the
     302             :   correct target architecture is likely to be `multicore-linux64`
     303             :   (or `multicore-darwin-x86_64` on macOS).
     304             :   On an HPC system, the correct Charm++ target architecture depends on the
     305             :   machine's inter-node communication architecture. We will be providing specific
     306             :   instructions for various HPC systems.
     307             : * Both Charm++ and SpECTRE must be compiled using the same compiler,
     308             :   otherwise you will receive undefined reference errors while linking SpECTRE.
     309             :   When compiling Charm++ you can specify the compiler using, for example,
     310             :   ```
     311             :   ./build charm++ ARCH clang
     312             :   ```
     313             :   When compiling SpECTRE you can specify the compiler to CMake using,
     314             :   for example,
     315             :   ```
     316             :   cmake -D CMAKE_CXX_COMPILER=clang++ \
     317             :         -D CMAKE_C_COMPILER=clang \
     318             :         -D CMAKE_Fortran_COMPILER=gfortran \
     319             :         -D CHARM_ROOT=CHARM_DIR/ARCH_OPTS SPECTRE_ROOT
     320             :   ```
     321             : * Inside the SpECTRE build directory, use `make list` to see all available
     322             :   targets. This list can be refreshed by running CMake again.
